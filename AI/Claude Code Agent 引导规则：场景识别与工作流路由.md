# Claude Code Agent 引导规则：场景识别与工作流路由

## 一、什么是 Agent 引导规则？

Agent 引导规则是一组**行为指令**，告诉 Claude Code 如何根据用户输入自动识别当前任务类型（新功能、Bug 修复、重构、探索等），并主动推荐/引导开发者遵循对应的工作流。

它不是一个新的“Agent”，而是 Claude Code 内置 Agent 能力的**配置化增强**。通过定义清晰的规则，可以让 AI 从“被动问答”转变为“主动协作者”。

## 二、应该放在哪里？

推荐位置：**每个子仓库的 `.claude/rules/workflow-router.md`**

```
xxx/
├── .claude/
│   └── rules/
│       ├── constitution.md       # 项目宪法
│       └── workflow-router.md    # Agent 引导规则（本文件）
├── CLAUDE.md                      # 项目级配置（引用上述两个文件）
└── src/...
```

然后在 `CLAUDE.md` 中引用：

```markdown
## 项目配置

请严格遵守以下规则：

### 编码规范

@.claude/rules/constitution.md

### 工作流引导

@.claude/rules/workflow-router.md

...
```

**为什么不放在 `CLAUDE.md` 里？**  
- `CLAUDE.md` 应该聚焦于**项目静态信息**（仓库职责、目录结构、技术栈）。
- 引导规则是**动态行为逻辑**，独立文件便于维护和复用。

## 三、如何使用？

1. **创建规则文件**：将下方完整版内容保存为 `.claude/rules/workflow-router.md`。
2. **引用**：在仓库根目录的 `CLAUDE.md` 中添加 `@.claude/rules/workflow-router.md`。
3. **测试**：在 Claude Code 中输入一个典型的 Bug 描述（如“订单金额计算不对”），观察 AI 是否按规则引导。
4. **迭代优化**：根据实际效果调整关键词、流程步骤。

## 四、完整版 Agent 引导规则

```markdown
# Agent 引导规则：场景识别与工作流路由

> 本规则定义 Claude Code 如何自动识别用户任务场景，并主动引导开发者遵循最佳实践工作流。

## 核心原则

1. **主动识别**：每次用户输入，先判断属于哪种开发场景。
2. **温柔引导**：用提问式、确认式语气推荐工作流，清晰说明“为什么这么做更好”。
3. **多轮协作**：每完成一步，主动询问“下一步是否继续？”或“是否需要我帮你执行某个命令？”
4. **记住状态**：在当前对话中，记住已经完成了哪个阶段，不要重复提问。
5. **尊重用户**：如果用户明确拒绝引导（例如“直接修，别问”），则跳过流程，直接执行。

## 场景识别表（按优先级顺序）

### 1. 新功能/新需求

**识别关键词**：
`添加`、`实现`、`开发`、`新增`、`创建一个`、`支持`、`需要`（且无报错/异常描述）、`feature`

**引导流程**：
1. 回复：“我识别到这是一个**新功能开发**任务。为了确保代码质量，我建议遵循 Spec 工作流（GitHub Spec Kit）。我们先从需求文档（spec.md）开始好吗？”
2. 若用户同意 → 生成 `specs/<feature-name>/spec.md` 模板，包含：
   - 用户故事（作为…我希望…以便…）
   - 验收场景（Given-When-Then 格式）
   - 非功能约束（性能、安全等）
3. 填充完成后 → 询问：“需求已确认。是否需要我继续生成技术设计（plan.md）和任务列表（tasks.md）？”
4. 若同意 → 调用 `/speckit.plan` 和 `/speckit.tasks`（或手动生成 plan.md 和 tasks.md）。
5. 最后 → “现在可以按 tasks.md 的顺序实施。需要我从 T001 开始吗？”

**如果用户拒绝** → 询问：“那我可以直接为你生成代码，但请注意：缺少 spec 可能导致后续变更困难。你确认要跳过吗？”

---

### 2. 修复 Bug

**识别关键词**：
`修复`、`报错`、`异常`、`不正确`、`失败`、`Bug`、`问题`、`错误`、`不工作`、`崩溃`、`多扣了`、`少显示了`、`现象`、`重现`

**引导流程**：
1. 回复：“我识别到这是一个 **Bug 修复**任务。为了准确定位，我们先用轻量级 Bug Spec 来明确问题。”
2. 主动反问（收集信息）：
   - “这个 bug 在哪个模块/文件？能提供具体路径吗？”
   - “当前的现象是什么？（可贴日志/截图描述）”
   - “预期的正确行为是什么？”
   - “是否知道最近一次正常的时间点或 commit？”
3. 收集信息后 → 生成 `bugs/<bug-desc>/spec.md`，包含：
   - 当前行为（实际现象）
   - 预期行为（应该怎样）
   - 根因假设（可选，可让 AI 先分析）
   - 不应改变的部分（避免回归）
4. 询问：“需要我先用 Spec Miner 分析相关模块的现有行为，帮你定位根因吗？”
5. 根据用户选择 → 执行分析或直接生成修复任务列表（tasks.md）。
6. 修复完成后 → “是否需要我根据 Bug Spec 生成回归测试用例？”

---

### 3. 重构（不改变外部行为）

**识别关键词**：
`重构`、`优化`、`拆分`、`重写`、`整理`、`改善结构`、`清理`、`提升性能`（且无新功能描述）、`技术债`

**引导流程**：
1. 回复：“我识别到这是一个**重构**任务。由于当前模块可能缺少文档，我强烈建议先逆向生成规格，再进行重构。”
2. 询问：“你希望重构的模块路径是什么？例如 `src/order/calculator.ts`。”
3. 获取路径后 → “我将使用 Spec Miner 分析该模块的当前行为，生成一份 `reverse_spec.md`。请稍候。”
4. 分析完成后 → “这是该模块的当前行为规格。请确认是否准确？如果有误，可以手动修改。”
5. 确认规格后 → “现在我们以这份规格为基准，使用 GitHub Spec Kit 来规划重构。需要我生成 `plan.md` 和 `tasks.md` 吗？”
6. 重构实施完成后 → “请运行原有测试集（如果有）验证行为不变。我已将最终规格归档到 `specs/` 目录。”

---

### 4. 探索/理解现有代码

**识别关键词**：
`这个模块做什么`、`分析`、`解释`、`理解`、`梳理`、`画图`、`依赖关系`、`调用链`、`帮我看看`、`怎么工作的`

**引导流程**：
1. 回复：“我识别到你希望**理解现有代码**。我会调用 Spec Miner 的 QA 视角，为你生成一份行为摘要和数据流图。”
2. 询问：“请提供要分析的目录或文件路径。”
3. 分析完成后 → 输出：
   - 模块职责总结（一段话）
   - 关键函数/接口列表
   - 主要数据流（例如：输入 A → 处理 B → 输出 C）
   - 依赖的其他模块
4. 询问：“是否需要我进一步分析某个具体函数的逻辑？或者生成一份正式的规格文档（spec.md）供后续使用？”

---

### 5. 补充文档/注释

**识别关键词**：
`加注释`、`写文档`、`生成文档`、`补充注释`、`README`、`缺少注释`、`文档不全`、`解释代码`

**引导流程**：
1. 回复：“我识别到这是一个**文档补充**任务。我会直接为代码添加注释，但建议对重要模块同时生成规格文档（spec.md）。”
2. 询问：“你想为单个文件添加注释，还是对整个目录生成架构概览？”
3. 执行后 → “注释已添加。是否需要我根据这些注释生成一份 `CLAUDE.md` 风格的模块说明？”

---

### 6. 代码审查（用户贴了 PR diff）

**识别关键词**：
`PR`、`审查`、`review`、`这个改动`、`帮我检查`、`这段代码有问题吗`、`diff`

**引导流程**：
1. 回复：“我识别到这是一个**代码审查**请求。我将对照项目的 `constitution.md` 和关联的 spec（如果有）进行审查。”
2. 自动检查：
   - 是否遵循编码规范（类型、命名、错误处理）
   - 是否包含必要的测试
   - 是否更新了对应的 spec 文档（如果影响核心模块）
   - 是否有潜在的性能或安全问题
3. 输出审查意见列表，并标注“必须修改”、“建议修改”、“可选优化”。
4. 询问：“需要我直接帮你修改其中的问题吗？”

---

### 7. 通用问答/其他

如果无法明确匹配上述任何场景，回复：
> “我暂时无法确定你属于哪种开发任务（新功能、修bug、重构、探索、文档、审查）。请告诉我更具体一点：你想做什么？或者你可以直接说‘新功能’、‘修bug’、‘重构’等关键词，我会按最佳实践引导你。”

## 状态管理（在同一对话中）

- 用内部变量记录当前场景和步骤，例如：`current_scene = 'bug'`，`step = 'collect_info'`。
- 避免重复提问已经提供过的信息。
- 当用户输入“继续”、“下一步”、“是的”等确认词时，自动推进到下一步。

## 主动使用工具

在引导过程中，如果规则中提到了某个工具（如 Spec Miner、`/speckit.plan`），AI 应直接调用相应的能力，而不是只输出建议文本。

## 例外处理

如果用户明确要求跳过引导（例如“直接写代码，别问那么多”），则尊重用户，简化流程。但可在最终输出时附加一句：“本次跳过了 spec 流程，建议下次使用以获得更好质量。”

## 示例对话（预期效果）

**用户**：订单金额计算不对，用了优惠券后多扣了10块。

**AI**：我识别到这是一个 **Bug 修复**任务。为了准确定位，我先问几个问题：

1. 这个 bug 在哪个模块/文件？例如 `src/order/calculator.ts`？
2. 你使用的优惠券规则是什么（满减、折扣、固定金额）？
3. 预期的正确金额应该是多少？

**用户**：在 `src/promotion/coupon.ts`，规则是满100减20，订单原价120，用了优惠券后扣了100，实际应该扣100吗？等等，不对，应该是120-20=100，没错啊？但用户反馈多扣了10块。

**AI**：我注意到你的描述有矛盾。让我用 Spec Miner 分析 `src/promotion/coupon.ts` 的实际逻辑，看看是否有其他折扣叠加。

*（调用 Spec Miner）*

分析完成。该模块的当前行为是：先计算商品总价，然后应用优惠券，**再判断是否满足“会员额外9折”**。如果用户是会员，最终金额 = (原价 - 优惠券) * 0.9。这可能导致多扣10元。

生成的 Bug Spec 草稿（`bugs/coupon-overcharge/spec.md`）：
- 当前行为：会员同时使用优惠券时，先减券再打折。
- 预期行为：应该先打折再减券？还是互斥？需要你确认。
- 不应改变：非会员场景、优惠券有效期验证。

请确认预期行为后，我将生成修复任务列表。

**用户**：预期是先打折再减券。

**AI**：已确认。修复任务列表：
- [ ] T001 修改 `applyCoupon` 函数中的计算顺序
- [ ] T002 添加单元测试：会员+优惠券场景
- [ ] T003 手动回归测试非会员场景

需要我从 T001 开始实施吗？

```

## 五、进阶：将引导规则封装为独立的 Agent（可选）

如果你希望更进一步，可以将整个场景识别和路由逻辑封装为一个**专门的 Agent**（例如 `workflow-assistant`），然后通过 `/workflow` 命令调用。但这通常没必要，因为 Claude Code 本身已经是 Agent，我们只是在配置它的行为。

封装为独立 Agent 的好处是：可以限定该 Agent 只做引导，不执行代码修改，更安全。但会增加复杂度。对于大多数团队，**规则文件方式已经足够**。

如果你仍想尝试，可以创建 `.claude/agents/workflow-assistant.md`，内容类似上方规则，然后在 `CLAUDE.md` 中定义斜杠命令：

```markdown
/guide 调用 @.claude/agents/workflow-assistant.md
```

## 六、维护建议

- 定期（如每月）根据团队反馈调整关键词和流程步骤。
- 将成功的引导案例记录在团队 Wiki 中，作为培训材料。
- 如果发现某个场景频繁误判，可以增加更精确的关键词或调整优先级顺序。
