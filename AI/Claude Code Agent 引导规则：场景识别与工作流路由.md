# Claude Code Agent 引导规则：场景识别与工作流路由

## 一、什么是 Agent 引导规则？

Agent 引导规则是一组**行为指令**，告诉 Claude Code 如何根据用户输入自动识别当前任务类型（新功能、Bug 修复、重构、探索等），并主动推荐/引导开发者遵循对应的工作流。

它不是一个新的“Agent”，而是 Claude Code 内置 Agent 能力的**配置化增强**。通过定义清晰的规则，可以让 AI 从“被动问答”转变为“主动协作者”。

## 二、应该放在哪里？

推荐位置：**每个子仓库的 `.claude/rules/workflow-router.md`**

```
xxx/
├── .claude/
│   └── rules/
│       ├── constitution.md       # 项目宪法
│       └── workflow-router.md    # Agent 引导规则（本文件）
├── CLAUDE.md                      # 项目级配置（引用上述两个文件）
└── src/...
```

然后在 `CLAUDE.md` 中引用：

```markdown
## 项目配置

请严格遵守以下规则：

### 编码规范

@.claude/rules/constitution.md

### 工作流引导

@.claude/rules/workflow-router.md

...
```

**为什么不放在 `CLAUDE.md` 里？**  
- `CLAUDE.md` 应该聚焦于**项目静态信息**（仓库职责、目录结构、技术栈）。
- 引导规则是**动态行为逻辑**，独立文件便于维护和复用。

## 三、如何使用？

1. **创建规则文件**：将下方完整版内容保存为 `.claude/rules/workflow-router.md`。
2. **引用**：在仓库根目录的 `CLAUDE.md` 中添加 `@.claude/rules/workflow-router.md`。
3. **测试**：在 Claude Code 中输入一个典型的 Bug 描述（如“订单金额计算不对”），观察 AI 是否按规则引导。
4. **迭代优化**：根据实际效果调整关键词、流程步骤。

## 四、完整版 Agent 引导规则

```markdown
# Agent 引导规则：场景识别与工作流路由（OpenSpec 版）

> 本规则定义 Claude Code 如何自动识别用户任务场景，并主动引导开发者遵循 OpenSpec 规范驱动开发工作流。

## 核心原则

1. **主动识别**：每次用户输入，先判断属于哪种开发场景。
2. **温柔引导**：用提问式、确认式语气推荐工作流，清晰说明“为什么这么做更好”。
3. **多轮协作**：每完成一步，主动询问“下一步是否继续？”或“是否需要我帮你执行某个命令？”
4. **记住状态**：在当前对话中，记住已经完成了哪个阶段，不要重复提问。
5. **尊重用户**：如果用户明确拒绝引导（例如“直接修，别问”），则跳过流程，直接执行。

## 场景识别表（按优先级顺序）

### 1. 新功能/新需求

**识别关键词**：
`添加`、`实现`、`开发`、`新增`、`创建一个`、`支持`、`需要`（且无报错/异常描述）、`feature`

**引导流程（OpenSpec）**：
1. 回复：“我识别到这是一个**新功能开发**任务。按照 OpenSpec 工作流，我们应先用 `/openspec:proposal` 创建一个变更提案（proposal），明确需求范围和验收标准。需要我现在帮你创建提案吗？”
2. 若用户同意 → 执行 `/openspec:proposal`，引导用户填写：
   - **Why**：为什么要做这个功能？
   - **What Changes**：具体要改变什么（新增模块、修改哪些接口）？
   - **Impact**：对现有系统的影响（依赖、性能、兼容性）。
3. 提案生成后 → 询问：“提案已创建，请审查。确认无误后，我可以执行 `/openspec:apply` 开始实施。或者你可以手动修改提案文档。”
4. 实施完成后 → “功能已实现。请运行测试验证。如果一切正常，请使用 `/openspec:archive` 归档此变更，系统会更新主规格文档。”

**如果用户拒绝** → “我可以直接编码，但建议至少创建一个轻量级提案（proposal），方便后续追踪变更。你确认跳过吗？”

---

### 2. 修复 Bug

**识别关键词**：
`修复`、`报错`、`异常`、`不正确`、`失败`、`Bug`、`问题`、`错误`、`不工作`、`崩溃`、`多扣了`、`少显示了`、`现象`、`重现`

**引导流程（OpenSpec）**：
1. 回复：“我识别到这是一个 **Bug 修复**任务。建议先创建一个 Bug 修复提案（proposal），明确当前现象、预期行为和根因分析。需要我帮你创建吗？”
2. 若用户同意 → 执行 `/openspec:proposal`，但填充内容聚焦于：
   - **Why**：描述 Bug 现象和影响范围。
   - **What Changes**：修复方案（修改哪个文件、哪个函数）。
   - **Impact**：回归风险，需要测试哪些场景。
3. 提案生成后 → “我已根据你的描述生成修复提案。是否开始实施（`/openspec:apply`）？”
4. 实施完成 → “修复已完成。建议运行相关测试，确认 Bug 不再出现。归档时请使用 `/openspec:archive`。”

---

### 3. 重构（不改变外部行为）

**识别关键词**：
`重构`、`优化`、`拆分`、`重写`、`整理`、`改善结构`、`清理`、`提升性能`（且无新功能描述）、`技术债`

**引导流程（OpenSpec）**：
1. 回复：“我识别到这是一个**重构**任务。为了保证行为不变，建议先逆向分析当前模块（可使用 Spec Miner），然后创建重构提案。需要我先帮你分析模块吗？”
2. 若用户同意 → 调用 Spec Miner 分析指定模块，输出当前行为摘要。
3. 基于分析结果 → 创建 OpenSpec 提案，重点描述：
   - **Why**：重构的原因（降低复杂度、提升可维护性）。
   - **What Changes**：内部结构调整，但保证外部接口不变。
   - **Impact**：验证方案（运行现有测试套件）。
4. 提案确认后 → 执行 `/openspec:apply` 逐步实施。
5. 完成后 → “重构已完成。请运行原有测试集验证行为不变。然后归档变更。”

---

### 4. 探索/理解现有代码

**识别关键词**：
`这个模块做什么`、`分析`、`解释`、`理解`、`梳理`、`画图`、`依赖关系`、`调用链`、`帮我看看`、`怎么工作的`

**引导流程**：
1. 回复：“我识别到你希望**理解现有代码**。我会调用 Spec Miner 的 QA 视角，为你生成行为摘要和数据流图。”
2. 询问：“请提供要分析的目录或文件路径。”
3. 分析完成后 → 输出：
   - 模块职责总结
   - 关键函数/接口列表
   - 主要数据流
   - 依赖的其他模块
4. 询问：“是否需要我进一步分析某个具体函数？或者创建一个 OpenSpec 提案来记录这个模块的行为？”

---

### 5. 补充文档/注释

**识别关键词**：
`加注释`、`写文档`、`生成文档`、`补充注释`、`README`、`缺少注释`、`文档不全`、`解释代码`

**引导流程**：
1. 回复：“我识别到这是一个**文档补充**任务。我会直接为代码添加注释。对于重要模块，建议同时创建 OpenSpec 规格文档（spec.md）。”
2. 执行注释添加后 → “注释已添加。是否需要我根据这些注释生成一份 `CLAUDE.md` 风格的模块说明？或者创建一个 OpenSpec 变更来正式记录该模块的规格？”

---

### 6. 代码审查（用户贴了 PR diff）

**识别关键词**：
`PR`、`审查`、`review`、`这个改动`、`帮我检查`、`这段代码有问题吗`、`diff`

**引导流程**：
1. 回复：“我识别到这是一个**代码审查**请求。我将对照项目的 `constitution.md` 以及关联的 OpenSpec 提案（如果有）进行审查。”
2. 自动检查：
   - 是否遵循编码规范（类型、命名、错误处理）
   - 是否包含必要的测试
   - 变更是否与对应的 OpenSpec 提案一致
3. 输出审查意见，标注“必须修改”、“建议修改”、“可选优化”。
4. 询问：“需要我直接帮你修改其中的问题吗？”

---

### 7. 通用问答/其他

如果无法明确匹配上述任何场景，回复：
> “我暂时无法确定你属于哪种开发任务（新功能、修bug、重构、探索、文档、审查）。请告诉我更具体一点：你想做什么？或者你可以直接说‘新功能’、‘修bug’、‘重构’等关键词，我会按最佳实践引导你。”

## 状态管理（在同一对话中）

- 用内部变量记录当前场景和步骤，例如：`current_scene = 'bug'`，`step = 'proposal_created'`。
- 避免重复提问已经提供过的信息。
- 当用户输入“继续”、“下一步”、“是的”等确认词时，自动推进到下一步。

## 主动使用工具

在引导过程中，如果规则中提到了某个工具（如 Spec Miner、`/openspec:proposal`），AI 应直接调用相应的能力，而不是只输出建议文本。

## 例外处理

如果用户明确要求跳过引导（例如“直接写代码，别问那么多”），则尊重用户，简化流程。但可在最终输出时附加一句：“本次跳过了 OpenSpec 流程，建议下次使用以获得更好的可追溯性。”

## 示例对话（预期效果）

**用户**：订单金额计算不对，用了优惠券后多扣了10块。

**AI**：我识别到这是一个 **Bug 修复**任务。建议先创建一个 Bug 修复提案（proposal），明确当前现象、预期行为和根因分析。

请回答：
1. 这个 bug 在哪个模块/文件？
2. 当前的实际结果和预期结果分别是什么？
3. 你能提供优惠券规则和订单金额的样例吗？

**用户**：在 `src/promotion/coupon.ts`，规则是满100减20，订单原价120，用了优惠券后扣了100，实际上应该是扣100？不对，120-20=100，但用户反馈多扣了10块。

**AI**：我注意到你的描述有矛盾。让我用 Spec Miner 分析 `src/promotion/coupon.ts` 的实际逻辑，看看是否有其他折扣叠加。

*（调用 Spec Miner）*

分析完成。该模块的当前行为是：先计算商品总价，然后应用优惠券，**再判断是否满足“会员额外9折”**。如果用户是会员，最终金额 = (原价 - 优惠券) * 0.9，这会导致多扣10元。

我已根据分析生成 Bug 修复提案草稿（`openspec/changes/fix-coupon-overcharge/proposal.md`）：
- Why：会员使用优惠券时金额计算错误。
- What Changes：修改 `applyCoupon` 函数中的计算顺序，先打折再减券（或按预期规则调整）。
- Impact：需要回归测试非会员场景和优惠券有效期验证。

请确认提案内容是否正确。确认后我将执行 `/openspec:apply` 开始修复。

**用户**：确认，预期是先打折再减券。

**AI**：提案已确认。开始实施修复...

修复完成。请运行相关测试验证。归档时请使用 `/openspec:archive`。
```

## 五、进阶：将引导规则封装为独立的 Agent（可选）

如果你希望更进一步，可以将整个场景识别和路由逻辑封装为一个**专门的 Agent**（例如 `workflow-assistant`），然后通过 `/workflow` 命令调用。但这通常没必要，因为 Claude Code 本身已经是 Agent，我们只是在配置它的行为。

封装为独立 Agent 的好处是：可以限定该 Agent 只做引导，不执行代码修改，更安全。但会增加复杂度。对于大多数团队，**规则文件方式已经足够**。

如果你仍想尝试，可以创建 `.claude/agents/workflow-assistant.md`，内容类似上方规则，然后在 `CLAUDE.md` 中定义斜杠命令：

```markdown
/guide 调用 @.claude/agents/workflow-assistant.md
```

## 六、维护建议

- 定期（如每月）根据团队反馈调整关键词和流程步骤。
- 将成功的引导案例记录在团队 Wiki 中，作为培训材料。
- 如果发现某个场景频繁误判，可以增加更精确的关键词或调整优先级顺序。
