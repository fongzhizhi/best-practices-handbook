## 理解Commands、Skills、Agent

**一、三个概念的内涵**

| 概念         | 核心定义                                                     | 本质           | 类比                 |
| ------------ | ------------------------------------------------------------ | -------------- | -------------------- |
| **Commands** | 以 `/` 开头的快捷指令，用于触发一个预定义操作或启动一个工作流 | 一次性执行脚本 | IDE 快捷键 / 菜单项  |
| **Skills**   | 预先定义的“专业能力包”，加载后可改变 AI 的行为模式、知识范围或输出风格 | 领域专家模式   | 给 AI 换上“职业套装” |
| **Agent**    | 能够自主规划、调用工具、执行多步骤任务的智能体，最终完成一个复杂目标 | 自主执行单元   | 数字员工             |

---

**二、详细区别**

| 维度                     | Commands                                       | Skills                                                       | Agent                                                        |
| ------------------------ | ---------------------------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **触发方式**             | 用户在对话中输入 `/command` 触发               | 在对话中通过指令激活（如“使用 code-review skill”），或通过配置自动加载 | 通常由 `/agent` 命令或某些命令（如 `/spec-execute`）内部启动 |
| **作用范围**             | 单次执行或启动一个多步流程，完成后即结束       | 持续影响当前对话的 AI 行为，直到被替换或重置                 | 独立执行一个任务，可长时间运行，直到目标达成                 |
| **核心能力**             | 执行固定逻辑（如生成测试、创建文件、调用 API） | 注入特定领域的知识、规范、示例，改变 AI 的输出质量           | 自主分解目标、调用工具（读写文件、运行终端命令、Git操作等）、处理中间结果 |
| **是否需要用户持续参与** | 执行过程可能需要用户交互（如确认步骤）         | 加载后无需持续交互，AI 自动遵循规则                          | 自主运行，仅关键决策点（如危险操作）需用户确认               |
| **可编程性**             | 通过 Markdown 文件（含 YAML frontmatter）定义  | 通过 Markdown 文件（含 YAML frontmatter）定义，可定义指令、示例、规则 | 通过工作流配置、MCP 服务器或自定义 agent 脚本定义            |
| **典型用途**             | 快速执行重复性任务、启动规范工作流             | 统一团队代码风格、让 AI 精通特定技术栈                       | 多仓库重构、复杂 bug 分析、批量生成文档                      |

---

**三、最佳实践**

**Commands**

- **场景**：高频重复动作，如生成单元测试、创建新组件、添加 JSDoc、启动 Spec 工作流。
- **原则**：命令应聚焦单一职责，参数化以增加灵活性。
- **实践**：将团队常用的工作流封装成命令，提交到 `.claude/commands/` 供所有人使用。

**Skills**

- **场景**：需要 AI 以特定风格或领域知识回答问题，如代码审查、前端组件设计、数据库建模。
- **原则**：Skill 应提供具体规则、示例和反例，而非模糊描述。
- **实践**：创建团队级 Skill（如 `team-code-style`），让 AI 自动遵循项目规范。

**Agent**

- **场景**：跨文件/跨仓库的复杂任务，如重构核心模块、分析性能瓶颈、批量更新依赖。
- **原则**：明确目标，提供边界约束（如“只修改 src/ 目录下的文件”），必要时设置确认点。
- **实践**：使用 Spec 工作流时，由 Agent 自动执行任务清单；对于一次性复杂任务，直接启动 `/agent` 并描述目标。

**四、如何使用官方提供的能力**

1. **Commands**

**内置常用命令**（Claude Code 自带）：
- `/explain` – 解释选中的代码
- `/test` – 为当前文件生成测试
- `/fix` – 修复选中代码的问题
- `/doc` – 添加文档注释

**Spec 工作流添加的命令**（安装 `@pimzino/claude-code-spec-workflow` 后）：
- `/spec-create` – 创建新规范
- `/spec-requirements` – 编写需求
- `/spec-design` – 编写设计
- `/spec-tasks` – 生成任务清单
- `/spec-execute` – 执行任务

**使用方式**：在对话中直接输入 `/命令名` 即可，部分命令支持参数（如 `/spec-create 用户认证`）。

2. **Skills**

**官方内置 Skills**（部分）：
- `frontend-design` – 前端设计专家
- `code-review` – 代码审查专家
- `debugging` – 调试专家

**激活 Skill**：
```
# 在对话中直接说明
使用 code-review skill 审查我刚写的代码

# 或通过 /skill 命令（如果支持）
/skill code-review
```

**自动加载 Skill**：可以在 `.claude/skills/` 中放置团队 Skill，并在 `CLAUDE.md` 中配置默认加载。

**3. Agent**

**启用 Agent**：
- 直接输入 `/agent` 进入 Agent 模式，然后描述你的任务。
- 或在 Spec 工作流中，`/spec-execute` 会自动启动 Agent 来执行任务。

**Agent 能力**：可以调用文件读写、终端命令、Git 操作、网络请求等工具。执行过程中你可以在对话中干预或调整方向。

**五、如何创建自定义**

1. 自定义 **Command**

**步骤**：
1. 在项目根目录创建 `.claude/commands/` 文件夹（如果不存在）。
2. 创建一个 Markdown 文件，例如 `my-test.md`。
3. 在文件开头使用 YAML frontmatter 定义元数据，正文编写指令内容。

**示例：创建 `/my-test` 命令，为当前文件生成单元测试**
```markdown
---
name: my-test
description: 为当前打开的文件生成单元测试
---

请为当前文件中的每个公共函数生成单元测试，使用 Jest 框架，覆盖正常路径和边界条件。如果文件是 TypeScript，请确保类型正确。
测试文件命名规则：原文件名.test.js 或 .test.ts
```

**使用**：在 Claude Code 中输入 `/my-test`，AI 会按照你的描述执行。

2. **自定义** Skill

**步骤**：
1. 在项目根目录创建 `.claude/skills/` 文件夹。
2. 创建 Markdown 文件，例如 `team-code-style.md`。
3. 在 frontmatter 中定义技能名称和描述，正文写具体规则。

**示例：创建团队代码风格 Skill**
```markdown
---
name: team-code-style
description: 遵循公司代码规范，包括命名、注释、错误处理等
---

# 代码规范

## 命名
- 变量和函数：camelCase
- 类和接口：PascalCase
- 常量：UPPER_SNAKE_CASE

## 注释
- 所有公共函数必须有 JSDoc，描述参数和返回值
- 复杂逻辑必须添加行内注释解释为什么这样实现

## 错误处理
- 使用 try-catch 捕获异常，并记录日志
- 返回统一的错误响应格式：{ success: false, message: string, code: number }

## 示例
- 好的命名：`getUserById`
- 不好的命名：`get_user_by_id`
```

**激活 Skill**：

```
使用 team-code-style skill 帮我重构这个函数
```

3. **自定义** Agent

自定义 Agent 通常通过 **MCP (Model Context Protocol)** 实现，你可以为 Claude Code 添加外部工具或定义专门的 Agent 行为。

**简单方式**：通过 Spec 工作流的任务清单间接控制 Agent 行为（因为 `/spec-execute` 内部会启动 Agent）。你只需定义清晰的任务描述，Agent 会自主完成。

**高级方式**：编写 MCP 服务器（Node.js/Python），然后添加到 Claude Code：
```bash
claude mcp add my-agent npx my-agent-package@latest
```

之后 Claude Code 会自动识别 `my-agent` 工具，你可以通过自然语言调用它。例如：
```
使用 my-agent 帮我分析项目依赖关系并生成报告
```

**注意**：自定义 Agent 需要一定的开发能力，但对于大型项目非常实用，可以将重复性的复杂任务自动化。

**六、总结：何时使用哪个？**

| 任务类型                       | 推荐使用                                     |
| ------------------------------ | -------------------------------------------- |
| 快速生成测试、添加注释         | Command                                      |
| 需要统一输出风格（如代码规范） | Skill                                        |
| 多步骤、跨文件、需自主规划     | Agent                                        |
| 规范驱动开发新功能             | Spec 工作流（Commands + Skill + Agent 组合） |

通过灵活组合这三者，你可以将 Claude Code 从简单的对话助手升级为贴合团队工作流的高效工程伙伴。



## 使用 spec 工作流

**spec模式在Claude Code中的定位**

spec模式本质上就是Spec-Driven Development（SDD，规范驱动开发）范式，在Claude Code中对应的是“Subagent + Command”的组合实现。

| Kiro Spec模式              | Claude Code中的实现                             |
| :------------------------- | :---------------------------------------------- |
| 需求文档 (requirements.md) | `/spec-requirements` Command + 需求分析Subagent |
| 设计文档 (design.md)       | `/spec-design` Command + 架构设计Subagent       |
| 任务清单 (tasks.md)        | `/spec-tasks` Command + 任务拆解Subagent        |
| 逐步执行                   | `/spec-execute` Command + 实施Subagent          |

vibe Coding（氛围编程）最大的问题——不确定性：**从“碰运气”到“可控工程”**。

Spec Coding解决了Vibe Coding（氛围编程）最大的问题——不确定性：

- **Vibe Coding**：给个模糊需求，AI随机输出，像“拉霸游戏”一样看运气
- **Spec Coding**：先明确规范→再设计方案→再拆解任务→最后实现，每一步都要你确认

Spec Coding的核心机制是：**需求确认完才能进设计，设计确认完才能进任务，任务确认完才能写代码**。

这正好解决了你作为程序员最头疼的问题——AI写了一堆代码，但跟你要的完全不是一回事。

每个Spec都是一个独立的文件夹，包含需求、设计、任务三个核心文档：

- 一周后回来看，你能清楚知道“当时为什么这么设计”
- 出bug了能追溯到是哪个需求没覆盖到
- 团队成员可以基于同一份Spec协作

**强烈推荐使用的场景 ✅**

| 场景                   | 原因                                                     |
| :--------------------- | :------------------------------------------------------- |
| **新项目从0到1**       | 需要明确架构、技术选型、模块划分，Spec流程天然适合       |
| **新功能开发**         | 尤其是涉及多个文件、多个模块的功能，先设计后编码避免返工 |
| **团队协作项目**       | Spec文档就是“契约”，AI和人都能基于同一份规范工作         |
| **需要长期维护的项目** | 可追溯性让你三个月后还能理解当初的设计意图               |

**可以简化使用的场景 ⚠️**

| 场景             | 建议                                                    |
| :--------------- | :------------------------------------------------------ |
| **单文件小修改** | 直接对话模式更快，没必要走完整Spec流程                  |
| **探索性编码**   | 不确定方案时，先vibe coding快速试错，确定了再用Spec固化 |
| **紧急Bug修复**  | 直接改，别浪费时间写文档                                |

**一个实用的判断标准**

如果这个任务你需要花5分钟以上跟AI“解释清楚要做什么”，就值得用Spec模式。

**需要留意的坑**

1. 避免“过度文档化”

   Spec Coding的极端是“文档写半天，代码一行没写”，反而丧失了AI的效率优势。把握好度：**文档写到能让AI理解就行，不用追求完美格式**。

2. 适时人工干预

   Kiro官方评测中提到，即使有Spec，AI仍会犯错——比如生成有语法错误的代码、遗漏关键逻辑。在`/spec-execute`每个任务完成后，快速review一下代码，别完全放手。

3. 上下文窗口限制

   复杂项目的Spec文档可能很长，注意控制每个文档的长度，或使用渐进式披露（把详细内容拆到子文件）。

社区已经有完整的开源方案可以直接在Claude Code中使用这套工作流，以下是推荐的使用方式：

**一、传统方式：使用 `/spec-*` Slash Commands**

这种方式会在 Claude Code 中添加一系列 `/spec-` 开头的快捷命令，适合习惯命令驱动的用户。

**安装**

```bash
# 全局安装官方 Spec 工作流包
npm install -g @pimzino/claude-code-spec-workflow

# 在项目根目录执行初始化（会在当前目录生成 .claude/ 文件夹）
claude-code-spec-workflow
```

**使用**

启动 Claude Code 后，可以使用以下命令：

| 命令                         | 作用                               |
| ---------------------------- | ---------------------------------- |
| `/spec-create <名称> <描述>` | 创建一个新的 Spec（规范）          |
| `/spec-requirements`         | 编写需求文档（在已创建的 Spec 中） |
| `/spec-design`               | 编写技术设计文档                   |
| `/spec-tasks`                | 拆解任务清单                       |
| `/spec-execute <任务ID>`     | 执行指定的任务（可多步连续执行）   |
| `/spec-list`                 | 列出当前所有 Spec 及进度           |

**示例**：

```bash
# 在 Claude Code 对话中
/spec-create 用户认证 "实现基于 JWT 的用户登录和注册功能"
# 按提示依次生成需求、设计、任务
/spec-execute 1   # 执行第一个任务
```

**可选配置**

- 可在 `.claude/config.json` 中调整工作流细节（如是否自动生成测试、是否要求人工确认等）。

---

**二、MCP 方式：实时仪表盘 + 自然语言控制**

MCP 方式将 Spec 工作流集成到 Claude Code 的 **Model Context Protocol** 中，可通过仪表盘可视化管理，同时支持自然语言交互。

**添加 MCP 服务器**

```bash
# 在任意目录执行以下命令，将 MCP 服务器添加到 Claude Code
claude mcp add spec-workflow npx @pimzino/spec-workflow-mcp@latest -- /path/to/your/project
```
- 注意将 `/path/to/your/project` 替换为实际项目路径（支持多个项目，后续可在对话中切换）。
- 如果希望全局生效，可以不加路径，之后在对话中指定项目。

**启动仪表盘（可选，推荐）**

```bash
npx -y @pimzino/spec-workflow-mcp@latest --dashboard
```
仪表盘会在浏览器中打开，你可以：
- 查看所有 Spec 及其任务状态
- 通过界面创建/修改 Spec
- 实时跟踪执行进度

**在** Claude Code 中使用

安装 MCP 服务器后，Claude Code 会自动识别 `spec-workflow` 工具。你可以通过自然语言与之交互，例如：

```
请帮我为“订单退款功能”创建一个新的 Spec，并生成需求文档。
```

或者直接说：

```
查看当前项目的所有 Spec 进度。
```

Claude 会调用 MCP 工具完成操作，并在聊天中反馈结果。你也可以在仪表盘上手动操作，二者实时同步。

---

**三、两种方式对比**

| 特性       | 传统方式（Slash Commands） | MCP 方式                                 |
| ---------- | -------------------------- | ---------------------------------------- |
| 安装复杂度 | 简单（全局安装+初始化）    | 中等（需配置 MCP 服务器）                |
| 交互方式   | 必须使用 `/spec-*` 命令    | 自然语言 + 可选仪表盘                    |
| 可视化     | 无                         | 有（仪表盘）                             |
| 多项目支持 | 通过切换工作目录           | 配置多个项目路径，对话中切换             |
| 推荐场景   | 单项目、习惯命令行的用户   | 多项目管理、偏好图形界面或自然语言的用户 |

**四、常见问题**

Q1：安装后 `/spec-create` 命令不可用？

- 确保已正确执行 `claude-code-spec-workflow` 初始化命令，并且 `.claude/commands/` 目录下生成了命令文件。
- 重启 Claude Code 后生效。

Q2：MCP 方式中如何切换项目？

- 在对话中说“切换到项目 /path/to/another-project”，MCP 服务器会自动重定向。
- 或在仪表板顶部选择项目。

Q3：两种方式可以共存吗？

- 可以。传统方式安装的命令文件和 MCP 服务器互不冲突。但注意不要在同一会话中同时用两种方式操作同一个 Spec，以免状态冲突。

**五、开始你的第一个 Spec**

**传统方式**：

```
/spec-create 用户中心 "用户资料修改、头像上传、密码重置"
```

**MCP 方式**：
```
帮我为“用户中心”创建一个 Spec，需求包括：修改资料、上传头像、重置密码。
```

之后按照提示逐步完成需求、设计、任务拆解，然后使用 `/spec-execute` 或自然语言逐步执行任务，每个任务完成后会自动生成代码并提示测试。

通过这两种方式，你可以将 AI 辅助开发从“随性对话”升级为“规范驱动”的工程化流程，显著提升大型项目的可维护性和代码质量。
