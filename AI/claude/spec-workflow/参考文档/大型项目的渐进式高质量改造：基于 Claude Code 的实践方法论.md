# 大型项目的渐进式高质量改造：基于 Claude Code 的实践方法论

> 事必有法，然后可成。在讨论方法之前，先要认清现状，明确目标。

## 一、背景与现状

- **项目类型**：大型多仓库（Monorepo / Multi-repo）TypeScript 项目。
- **代码规模**：数百万行代码，数十个活跃仓库。
- **文档情况**：严重缺乏系统文档，模块注释稀少。
- **测试情况**：几乎没有自动化测试，依赖人工验证。
- **团队规模**：数十名开发者并行协作。
- **可维护性**：持续下降，新功能开发缓慢，Bug 修复容易引入回归。

## 二、目标：渐进式构建高质量可维护项目

“渐进式”是因为项目体量巨大，无法推倒重来，只能在不中断业务的前提下逐步改造。

“高质量”可以量化为以下六个维度：

1. **代码质量与一致性**  
   - 统一的编码规范（命名、格式、错误处理、异步模式）  
   - TypeScript `strict` 模式，严格控制 `any`  
   - 低复杂度、高内聚、无重复逻辑

2. **架构与边界清晰**  
   - 模块/仓库责任明确，对外接口最小化  
   - 依赖关系可理解，无循环依赖  
   - 关键依赖通过抽象层隔离，便于替换和测试

3. **文档与知识可获取**  
   - 系统级架构概览（CLAUDE.md、README）  
   - 模块级规格说明（Spec）  
   - 重要决策记录（ADR）

4. **可测试性与自动化验证**  
   - 分层测试（单元、集成、端到端）  
   - 测试即规格，稳定且可读  
   - CI 强制门禁（类型检查、lint、测试、构建）

5. **协作与流程高效**  
   - 每个 PR 聚焦单一职责  
   - 审查可对照规格（Spec）  
   - 知识共享前置，减少返工  
   - 自动化辅助重复劳动

6. **可观测性与可调试性**  
   - 结构化日志，关键路径可追踪  
   - 统一的错误处理规范  
   - 性能基准与监控

“可维护”是以上维度的综合体现——一个系统如果代码混乱、架构模糊、无文档无测试，必然难以维护。

## 三、为什么选择 AI Coding 这个方向？

即使没有 AI，上述需求同样存在。但 AI 带来了三个独特价值：

- **快速试错**：AI 可以在几分钟内生成多种方案原型，帮助团队快速验证思路。
- **严守规则**：AI 比人更擅长机械地遵守规范（如命名、格式、测试覆盖），且不会因为疲劳而疏忽。
- **提高执行效率**：将重复性、模板化的编码工作交给 AI，让人工聚焦于设计、审查和决策。

然而，必须清醒认识到：**高质量代码只是“高质量项目”的一环**。没有清晰的架构、完备的文档、可靠的测试，即使每一行代码都写得漂亮，整个系统依然会腐化。AI 不应该替代人的思考，而应作为**执行者**和**协作者**。如果 AI 的执行效率或质量不如人工，那引入 AI 就没有意义——这正是我们需要验证和持续优化的命题。

## 四、工具选型

公司已决策使用 **Claude Code** 作为 AI 编程辅助工具。Claude Code 具备以下能力：

- 与本地文件系统交互（读、写、搜索）
- 执行终端命令
- 支持自定义指令（`CLAUDE.md`、`.claude/rules/`、`skills/`、`commands/`）
- 可集成 MCP（Model Context Protocol）扩展

因此，我们的核心问题变为：**如何利用 Claude Code，从上述六个维度出发，渐进式改造项目？**

## 五、基于六维度的工作流设计

### 维度一：代码质量与一致性

**现状痛点**：  
- 无统一编码规范，不同开发者风格迥异  
- TypeScript 类型注解大量缺失，`any` 泛滥  
- 复杂函数随处可见，重复逻辑多

**Claude Code 解决方案**：

1. **建立规范宪法（Constitution）**  
   - 创建 `.claude/rules/constitution.md`，明确定义：
     - 命名规范（变量、函数、类、文件）
     - 错误处理模式（`Result` 类型或 `try/catch` 规范）
     - 异步模式（`Promise` vs `async/await`）
     - TypeScript `strict` 要求，禁止 `any` 的例外清单
   - 将 `constitution.md` 写入 `CLAUDE.md` 的引用，让 Claude Code 自动遵循。

2. **批量类型补全**  
   - 编写专用命令 `/add-types`，让 Claude Code 逐文件添加类型注解，将 `any` 替换为具体类型。
   - 人工审查 PR，逐步开启 `tsconfig.json` 的 `strict` 标志。

3. **重复代码检测与消除**  
   - 使用 Claude Code + `grep`/`ast-grep` 扫描相似代码块，输出重构建议。
   - 对确认的重复逻辑，用 Claude Code 生成统一的工具函数并替换所有调用点。

### 维度二：架构与边界清晰

**现状痛点**：  
- 模块职责模糊，一个文件承担多种功能  
- 依赖关系混乱，循环依赖常见  
- 多仓库间接口无契约，协同困难

**Claude Code 解决方案**：

1. **逆向分析模块边界（Spec Miner）**  
   - 对关键模块运行 Spec Miner，输出当前模块的职责、输入输出、依赖关系。
   - 人工审视输出，识别职责过重或边界模糊的模块。

2. **规划重构（GitHub Spec Kit或Open Spec）**  
   - 以逆向 Spec 为基准，使用 `/speckit.plan` 设计新的模块划分。
   - 生成 `tasks.md`，按任务列表逐步拆分模块。

3. **多仓库依赖治理**  
   - 在根目录 `CLAUDE.md` 中描述各仓库的职责和依赖关系。
   - 使用 Claude Code 扫描 `import` 语句，生成依赖矩阵，识别循环依赖或跨层依赖。
   - 逐步修正，将公共类型/工具抽取为共享包（如 `@myorg/shared-types`）。

### 维度三：文档与知识可获取

**现状痛点**：  
- 几乎没有系统文档，新人上手困难  
- 代码注释严重缺失  
- 历史决策无记录

**Claude Code 解决方案**：

1. **自动生成 CLAUDE.md**  
   - 运行 `/init` 命令，让 Claude Code 扫描项目并生成初始 `CLAUDE.md`（包含架构概览、关键目录、常用命令）。
   - 团队补充修正，持续维护。

2. **按需生成模块规格（Spec）**  
   - 对于重要模块（如支付、订单、用户），使用 Spec Miner 逆向生成 `spec.md`。
   - 对于新功能或重大 Bug，强制要求编写轻量级 Spec（可借助 `/speckit.specify`）。

3. **补充代码注释**  
   - 使用 Claude Code 的 `/explain` 命令为复杂函数添加注释。
   - 创建自定义命令 `/doc-file`，让 AI 为整个文件生成 JSDoc 风格注释。

4. **记录架构决策（ADR）**  
   - 在 `.claude/rules/adr-template.md` 中放置 ADR 模板。
   - 每次重大技术决策后，让 Claude Code 根据讨论生成 ADR 草稿，人工完善后入库。

### 维度四：可测试性与自动化验证

**现状痛点**：  
- 几乎没有测试，修改代码如履薄冰  
- 代码可测试性差（依赖具体实现、全局状态）

**Claude Code 解决方案**：

1. **为现有代码生成测试**  
   - 选择核心模块，使用 Claude Code 读取源文件并生成单元测试（Jest/Vitest）。
   - 测试用例基于逆向 Spec 中的验收场景。
   - 人工调整测试框架和环境配置。

2. **强制测试门禁**  
   - 在 `constitution.md` 中规定：所有新代码必须包含测试。
   - 在 CI 中配置测试覆盖率检查（如 `jest --coverage`），不达标则 PR 无法合并。

3. **改善可测试性**  
   - 使用 Claude Code 识别难以测试的函数（如直接调用 API、操作 DOM），建议依赖注入或抽象接口。
   - 生成重构计划，逐步替换为可测试的版本。

### 维度五：协作与流程高效

**现状痛点**：  
- 开发流程依赖人工记忆，容易遗漏步骤（如写测试、更新文档）  
- PR 审查缺乏共同参照，争议多  
- 知识传递依赖口口相传

**Claude Code 解决方案**：

1. **AI Agent 引导工作流**  
   - 在 `CLAUDE.md` 中配置“场景识别与路由”规则（见附录）。
   - 当开发者输入需求、Bug 或重构意图时，AI 主动推荐对应的工作流（如先写 Spec、先逆向分析等）。
   - 减少流程遗忘，统一团队行为。

2. **Spec 作为审查基准**  
   - 每个 PR 必须关联一个 Spec（新功能）或 Bug Spec（修复）。
   - 审查者对照 Spec 检查实现是否完整、正确。
   - 若 Spec 过时，要求同步更新。

3. **自动化重复劳动**  
   - 创建 `/pr-description` 命令，让 AI 根据当前分支的变更自动生成 PR 描述（包含变更摘要、测试情况、Spec 链接）。
   - 创建 `/update-spec` 命令，根据代码变更自动更新对应的 Spec 文档。

### 维度六：可观测性与可调试性

**现状痛点**：  
- 日志随意打印，格式不统一  
- 错误处理缺失，异常信息被吞没  
- 无性能基准，变更后无法感知性能回退

**Claude Code 解决方案**：

1. **统一日志规范**  
   - 在 `constitution.md` 中定义日志级别、格式、必含字段（如 `requestId`, `userId`, `duration`）。
   - 使用 Claude Code 批量改造现有 `console.log` 为结构化日志（如使用 `pino` 或 `winston`）。

2. **错误处理规范化**  
   - 定义自定义错误类型（如 `ValidationError`, `NotFoundError`）。
   - 让 Claude Code 扫描 `try/catch` 块，建议添加错误上下文和统一上报。

3. **性能基准与监控**  
   - 为关键 API 编写性能测试脚本（如使用 `autocannon`）。
   - 将性能测试集成到 CI，对 PR 进行基准对比，超过阈值则警告。

## 六、渐进式改造路线图

由于项目巨大，必须分阶段、分模块推进，避免“大爆炸”式改造带来的风险。

### 第一阶段：基础设施与规范（1-2 个月）

- 编写 `constitution.md` 并提交入库。
- 在根目录和各子仓库创建 `CLAUDE.md`（运行 `/init` 并人工补充）。
- 配置 CI 基础门禁：`tsc --noEmit`、`eslint`、`prettier`。
- 安装并验证 Claude Code 工作环境（确保团队每人可用）。

### 第二阶段：核心模块文档化与测试（2-3 个月）

- 选择 2-3 个最核心的业务模块（如订单、支付、用户），运行 Spec Miner 生成逆向 Spec。
- 人工评审 Spec，修正后入库。
- 为这些模块生成单元测试，并设置覆盖率门槛（初期可较低，如 50%）。
- 将 Spec 驱动的开发流程用于这些模块的后续变更。

### 第三阶段：工作流自动化与推广（2-3 个月）

- 完善 `CLAUDE.md` 中的 Agent 引导规则，封装 `/newfeat`、`/fixbug`、`/refactor` 等斜杠命令。
- 选择 1-2 个活跃开发团队试点，收集反馈并优化。
- 建立 PR 审查清单，强制关联 Spec。
- 开始清理低质量代码（类型补全、消除重复逻辑）。

### 第四阶段：全面推广与持续优化（长期）

- 将成功经验固化到 `.claude/` 配置中，推广至所有仓库。
- 定期（如每季度）对核心模块重新运行 Spec Miner，检测文档与代码是否一致。
- 持续改进 CI 门禁（增加性能基准、漏洞扫描等）。
- 建立内部最佳实践分享机制，鼓励团队贡献新的 skills 和 commands。

## 七、风险与应对

| 风险                  | 应对措施                                                     |
| --------------------- | ------------------------------------------------------------ |
| 团队抵触新流程        | 渐进式引入，先小范围试点，用数据证明效率提升；强调 AI 是助手而非监督者 |
| AI 生成代码质量不稳定 | 强制人工审查，设置低信任阈值；持续优化提示词和规则           |
| Spec 文档与代码不同步 | 引入 `drift` 工具在 CI 中检查；培养“变更必改 spec”的习惯     |
| 多仓库配置同步难      | 将通用配置抽离为独立仓库，通过 submodule 或脚本同步          |
| 成本控制              | Claude Code 按量计费，鼓励开发者合理使用（如先设计后编码，减少反复试错） |

## 八、总结

本文档从“事必有法”的理念出发，先明确了现状与目标（高质量六维度），然后论证了为何选择 AI Coding 方向，并给出了基于 Claude Code 的具体工作流设计。核心思想是：

- **AI 是执行者，不是思考替代者**。人的职责是设计架构、制定规范、审查结果；AI 负责快速生成、严格遵循、自动验证。
- **渐进式改造**。从规范入手，再到核心模块文档化，最后全面推广。不追求一步到位，而是持续迭代。
- **六个维度相互支撑**。没有文档和测试，代码质量再高也无法维护；没有清晰的架构，代码和文档都会混乱。

最终目标不是“用 AI 写出完美代码”，而是“在 AI 的辅助下，让整个系统变得可理解、可测试、可演进”。这是一条漫长但值得投入的道路。

## 附录：示例 - CLAUDE.md 中的 Agent 引导规则（简化版）

```markdown
## AI Agent 行为规则

当用户输入时，自动识别场景并引导：

1. **新需求**（关键词：添加、实现、新增）  
   → “我识别到新功能开发，建议先写 Spec。需要我生成 spec.md 模板吗？”

2. **修 Bug**（关键词：修复、报错、异常）  
   → “请提供：当前现象、预期行为、涉及模块。需要我先用 Spec Miner 分析该模块吗？”

3. **重构**（关键词：重构、优化、拆分）  
   → “建议先逆向生成规格，再规划重构。请提供模块路径。”

4. **探索/理解**（关键词：分析、解释、这个模块做什么）  
   → “我将调用 Spec Miner 的 QA 模式，输出行为摘要。”

5. **默认** → “请告诉我这是新功能、修bug、重构还是理解代码？我会按最佳实践引导你。”
```

将此文件放置于项目根目录的 `CLAUDE.md` 中，即可让 Claude Code 自动遵循。
