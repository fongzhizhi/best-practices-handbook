# Vibe Coding 与 Spec Coding 编码风格说明指南

## 一、两种编码风格：从直觉到规范

在 AI 辅助编程时代，开发者的编码方式呈现出两种典型风格：**Vibe Coding** 和 **Spec Coding**。理解它们的差异，并学会在合适场景下灵活切换，是提升个人效率和团队协作质量的关键。

### 1.1 Vibe Coding —— 跟着感觉走

**定义**：Vibe Coding 是一种以**直觉和快速试错**驱动的编码方式。开发者通过与 AI 进行高带宽、口语化的对话，不断迭代代码，直到“感觉对了”为止。它不依赖预先写好的详细规范，而是依赖开发者的经验和即时反馈。

**典型行为**：

- “帮我写一个登录表单，要好看一点。”
- “这个按钮颜色不太对，改成蓝色试试。”
- “把这段代码优化一下，让它跑得更快。”

**优点**：
- **探索性强**：非常适合验证想法、搭建原型、快速试错。
- **门槛低**：不需要事先准备文档，想到哪说到哪。
- **流畅感**：开发者可以保持心流状态，不被文档打断。

**缺点**：
- **不可追溯**：几轮对话后，很难回溯“为什么要这样改”。
- **质量不稳定**：缺乏系统性思考，容易遗漏边界条件和异常处理。
- **协作困难**：其他同事接手时，完全不知道代码背后的设计意图。

### 1.2 Spec Coding —— 先想清楚再动手

**定义**：Spec Coding 是一种**规范驱动**的编码方式。在动手写代码之前，开发者必须先与 AI（或团队）就**需求范围、接口契约、验收标准**达成一致，形成一份轻量级规格文档（Spec）。之后的编码工作严格围绕这份 Spec 展开。

**典型行为**：
- “我们先明确一下这个功能的输入输出和边界条件。”
- “我已经创建了 OpenSpec 提案，请按照 `tasks.md` 逐步实现。”
- “这个变更修改了 API 返回结构，需要同步更新 Spec 文档。”

**优点**：

- **可追溯**：Spec 文档记录了“为什么做”和“怎么做”，是团队的活文档。
- **质量可控**：在编码前就思考清楚边界，减少返工和 Bug。
- **协作友好**：新人接手或 Code Review 时，对照 Spec 即可理解意图。
- **AI 执行更精准**：明确的规范让 AI 生成的代码更符合预期。

**缺点**：
- **前期成本略高**：需要花时间写 Spec，对于极简单的任务可能显得繁琐。
- **灵活性稍弱**：一旦 Spec 确定，中途变更需要先更新 Spec，流程稍长。

### 1.3 核心对比一览表

| 维度 | Vibe Coding | Spec Coding |
|:---|:---|:---|
| **驱动方式** | 直觉、对话、试错 | 规范、契约、验收标准 |
| **前期准备** | 无，直接开始 | 需要编写轻量级 Spec |
| **AI 交互模式** | 高带宽口语化指令 | 基于 Spec 的结构化任务执行 |
| **文档产出** | 极少或零散 | 生成并与代码同步的 Spec 文档 |
| **适用场景** | 原型验证、UI 微调、个人工具 | 功能开发、Bug 修复、重构、团队协作 |
| **可维护性** | 低，依赖个人记忆 | 高，Spec 即文档 |
| **协作成本** | 高，他人难以理解变更背景 | 低，Spec 提供统一上下文 |


## 二、何时用哪种？—— 场景化决策指南

两种风格并非水火不容，而是**互补关系**。聪明的开发者会根据任务阶段和性质灵活切换。

### 场景 1：从零探索一个新技术或新想法
- **推荐**：Vibe Coding
- **原因**：你还不确定最终形态，需要快速试错。先用 Vibe 跑通一个原型，理解核心逻辑。
- **示例**：“我想试试用 WebAssembly 处理图片，帮我搭一个最简单的 Demo。”

### 场景 2：在成熟项目中开发一个新功能
- **推荐**：Spec Coding
- **原因**：功能有明确的需求边界，需要与现有系统集成，必须保证质量和可维护性。
- **示例**：“我们要新增用户头像上传功能，支持裁剪和 OSS 存储。先用 `/openspec:proposal` 创建提案。”

### 场景 3：修复一个紧急的线上 Bug
- **推荐**：**Vibe 定位 + Spec 修复**
- **原因**：先用 Vibe Coding 的方式让 AI 帮你快速分析日志、定位问题模块；确定根因后，切换到 Spec Coding 创建修复提案，确保修复彻底且可追溯。
- **示例**：
  1. （Vibe）“线上登录报 500 错误，日志显示连接超时，帮我分析可能原因。”
  2. （Spec）“问题定位在 OSS 客户端超时设置。创建 Bug 修复提案，将超时从 10s 改为 30s，并增加重试机制。”

### 场景 4：重构一段祖传代码
- **推荐**：Spec Coding（配合 Spec Miner）
- **原因**：重构的前提是“不改变外部行为”。先用 Spec Miner 逆向提取现有代码的规范作为安全网，再创建 OpenSpec 提案规划重构步骤。
- **示例**：
  1. “用 spec-miner-openspec 分析 `src/legacy/order.ts`，提取当前行为规范。”
  2. “基于这份规范，创建重构提案，将巨型类拆分为三个服务类。”

### 场景 5：调整 UI 样式或文案
- **推荐**：Vibe Coding
- **原因**：变更范围极小，影响可控，快速迭代更高效。
- **示例**：“把登录按钮的颜色改成品牌蓝，圆角加大一点。”


## 三、在我们的工作流中如何结合？

我们的团队工作流设计已经为两种风格提供了**无缝切换的接口**：

| 你的意图 | 使用方式 | 背后风格 |
|:---|:---|:---|
| “帮我分析一下这个模块是干什么的” | 自然语言触发 Spec Miner | Vibe（探索）→ 产出 Spec |
| “我要开发一个新功能” | AI 引导执行 `/openspec:proposal` | Spec Coding |
| “这段代码帮我优化一下性能” | 直接描述，AI 修改后建议补充 Spec | Vibe 修改 → 补 Spec 固化 |
| “提交代码” | `/git-commit` 自动关联 ONES 和 Spec | Spec Coding（元数据追溯） |

**核心原则**：
- **探索用 Vibe，交付用 Spec**。
- **Vibe 的产出如果值得保留，就转化为 Spec**。
- **Spec 是团队协作的通用语言，Vibe 是个人效率的加速器**。


## 四、给团队的建议：从 Vibe 到 Spec 的渐进式习惯养成

我们理解完全切换到 Spec Coding 需要一个适应过程。建议团队按以下节奏逐步融入：

### 第一阶段：保持 Vibe，增加“事后记录”
- 当你用 Vibe Coding 完成一个任务后，花 1 分钟让 AI 帮你生成一段变更摘要，贴在 MR 描述里。
- 命令示例：“帮我总结一下刚才的改动，生成一段 MR 描述。”

### 第二阶段：关键任务强制 Spec
- 对于**核心业务逻辑、API 接口变更、复杂重构**，要求必须走 OpenSpec 流程。
- 其他任务仍可保持 Vibe 风格。

### 第三阶段：让 AI 帮你写 Spec
- 即使你习惯 Vibe Coding，也可以让 AI 根据你的对话上下文**自动生成一份 Spec 草稿**，你只需稍作修改即可。
- 命令示例：“根据我们刚才的讨论，帮我生成一份 OpenSpec 提案草稿。”

### 第四阶段：团队规范固化
- 将 Spec Coding 流程写入团队 Definition of Done：**“任何合并到主分支的变更，必须有对应的 Spec 文档或关联的 ONES 单。”**
- 通过 `/pre-commit` 和 CI 检查，让流程自动保障。


## 五、常见疑问解答

**Q：Spec Coding 会不会很慢？**
A：初期写 Spec 会多花几分钟，但这几分钟会在后续的 Code Review、测试、新人交接中节省数小时。长期来看，Spec Coding 反而**加速了整体交付效率**。

**Q：如果 Spec 写错了怎么办？**
A：Spec 是**活文档**，可以随时更新。OpenSpec 的 `proposal` → `apply` → `archive` 流程本身就支持迭代。发现 Spec 有误时，创建一个新的变更提案修正它即可。

**Q：小改动也要写 Spec 吗？**
A：不必。我们的规则是：**只要变更涉及接口、行为、业务逻辑，就需要 Spec；纯样式、文案、注释、配置调整可以跳过。** `/git-commit` 会自动识别并提示。

**Q：Vibe Coding 是不是不好？**
A：完全不是。Vibe Coding 是探索和创新的利器。我们反对的是**只用 Vibe 不用 Spec 导致的维护灾难**。最佳实践是**用 Vibe 探索，用 Spec 交付**。


## 六、总结

| 场景 | 推荐风格 | 一句话口诀 |
|:---|:---|:---|
| 探索新东西、搭原型 | Vibe Coding | “先跑起来再说。” |
| 开发功能、修复 Bug、重构 | Spec Coding | “先想清楚再动手。” |
| 调整样式、文案、配置 | Vibe Coding | “小修小改直接上。” |
| 分析旧代码、定位问题 | Vibe 探索 → Spec 固化 | “先看懂，再改造。” |

**最终目标**：让团队的每一行代码都有据可查，让 AI 成为可追溯、可信赖的协作伙伴。
