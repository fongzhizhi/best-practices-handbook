# 项目宪法constitution.md：概念、重要性与完整示例

## 一、什么是 constitution.md？

`constitution.md` 是项目的**最高行为准则**，一份短小、精炼、不可协商的原则清单。它不同于技术设计文档或用户手册，而是：

- **对团队**：统一认知，减少无谓争论（“宪法规定了，我们必须这样做”）。
- **对 AI**：作为 Claude Code 等 AI 编程助手的**系统提示**，在生成代码、审查 PR 时自动遵循。
- **对工具链**：CI 门禁、代码检查工具可以引用其中的条款，实现自动化验证。

**核心特征**：
- **不可协商**：除非全体维护者同意修订，否则必须遵守。
- **可验证**：每一条原则都能通过自动化工具或人工检查明确判断“是/否”。
- **渐进式**：允许对存量代码设置例外，但要求逐步收敛。

## 二、为什么需要 constitution.md？

在大型、多人协作的老旧项目中，没有明确的行为底线，会导致：

- 代码风格混乱，维护成本指数上升。
- 开发者凭个人偏好做决策，技术债持续累积。
- AI 生成的代码无法保持一致的质量标准。
- 代码审查缺乏共同依据，争论多、效率低。

**`constitution.md` 的价值**：

1. **降低认知负担**：不需要翻阅数十页规范文档，只需记住 8~12 条核心原则。
2. **对齐 AI 行为**：让 Claude Code 自动遵循团队约定，而非随机发挥。
3. **自动化门禁**：CI 可以直接检查“是否包含测试”、“是否通过 strict 编译”等宪法条款。
4. **知识传承**：新人入职后，阅读宪法能快速了解项目最重要的质量底线。

## 三、如何编写有效的 constitution.md？

### 3.1 内容来源

从“高质量项目的六个维度”中，各抽取 **1~2 条最基础、最紧急、最容易验证**的原则。避免贪多求全。

### 3.2 语言风格

- **祈使句**：直接写“必须……”、“禁止……”，而非“建议……”。
- **可量化**：使用“≥80%”、“必须包含”、“禁止使用”等明确词汇。
- **区分新旧**：明确规则适用范围（新代码 vs 存量代码）。

### 3.3 例外管理

老项目无法一步到位，需要设计例外机制：
- 允许在 `exceptions.json` 中列出暂时豁免的文件/模块，并附上技术债 ticket 和计划修复日期。
- 定期（如每季度）评审例外清单，强制清理。

### 3.4 存放与加载

- 存放路径：`.claude/rules/constitution.md`（因为 `rules/` 优先级高，可被路径限定）。

- 在根目录 `CLAUDE.md` 中引用：`@.claude/rules/constitution.md`。

  ```markdown
  ## 项目配置
  
  请严格遵守以下规则：
  
  ### 编码规范
  
  @.claude/rules/constitution.md
  
  ### 工作流引导
  
  @.claude/rules/workflow-router.md
  
  ...
  ```

- 同时将内容同步到团队 Wiki 或 README 中，方便人工查阅。

## 一、代码质量与一致性

### 1.1 TypeScript strict 模式

- **所有新编写或修改的代码**必须在 `tsconfig.json` 中开启 `strict: true` 的前提下通过编译。
- **存量代码**中的 `any` 必须标注 `// @ts-expect-error`，并附带原因及计划删除日期（格式：`// @ts-expect-error: 技术债 #1234, 计划 2026-06-01 前修复`）。
- **禁止**在新增代码中使用 `any`（包括函数参数、返回值、变量）。

### 1.2 命名规范

- 文件名：`kebab-case.ts`
- 类名、接口名、类型别名：`PascalCase`
- 变量名、函数名、参数名：`camelCase`
- 常量（`const` 声明且值不可变）：`UPPER_SNAKE_CASE`
- 私有成员：不加下划线前缀，依靠 TypeScript `private` 关键字

### 1.3 重复逻辑禁止

- 相同的业务规则或算法不得在两处及以上实现。
- AI 或人工发现重复后，必须抽取为共享函数或工具类，并替换所有调用点。

---

## 二、架构与边界清晰

### 2.1 模块边界

- 每个子仓库必须在根目录 `CLAUDE.md` 中声明其**对外暴露的公共 API**（导出的函数、类、类型）。
- **禁止**跨仓库直接导入其他仓库的 `src` 内部文件。必须通过公共 API 或共享包（`@myorg/shared-*`）访问。

### 2.2 依赖方向

- 依赖图不得出现循环（A → B → A）。
- 新增依赖前，必须运行 `npx madge --circular src/` 检查是否会引入循环。
- 依赖方向必须从具体到抽象（例如：业务模块 → 基础设施模块）。

---

## 三、文档与知识可获取

### 3.1 Spec 关联

- 任何影响**核心业务逻辑**的变更（新功能、重大 Bug 修复、重构）必须附带或更新 Spec 文档。
- Spec 文档存放于模块根目录的 `.specs/` 下，使用 Markdown 格式，至少包含：
  - 用户故事或问题描述
  - 验收场景（Given-When-Then）
  - 对现有行为的影响说明

### 3.2 CLAUDE.md 存在性

- 每个仓库根目录必须有一份 `CLAUDE.md`，描述：
  - 仓库职责（一句话）
  - 主要目录结构及作用
  - 常用开发命令（`npm run dev`、`npm test` 等）
  - 对外依赖的其他仓库或服务

## 四、可测试性与自动化验证

### 4.1 测试要求

- **所有新功能**必须包含至少一个单元测试（或集成测试，视模块性质而定）。
- **修复 Bug** 必须附带一个回归测试，该测试在修复前会失败、修复后通过。
- 测试覆盖率不得下降（以 `main` 分支为基准）。新增代码的行覆盖率应 ≥ 80%。

### 4.2 CI 门禁

- PR 合并前必须通过以下检查：
  - `tsc --noEmit`（TypeScript 类型检查）
  - `eslint`（使用项目配置）
  - `jest` 或对应测试框架（全部通过）
  - 构建（`npm run build`）成功
- 门禁失败时，PR 禁止合并，除非获得技术负责人特批。

---

## 五、协作与流程高效

### 5.1 PR 描述模板

- 每个 PR 的描述必须包含以下小节：

  ```markdown
  ## 变更摘要
  （简述改动内容）
  
  ## Spec 关联
  - [ ] 已更新/创建 Spec 文档（路径：`...`）
  - [ ] 不涉及核心逻辑，无需 Spec
  
  ## 测试情况
  - [ ] 单元测试通过
  - [ ] 手动测试（描述场景）
  
  ## 对核心模块的影响
  - [ ] 无影响
  - [ ] 有影响，已在 Spec 中说明
  
  ### 5.2 审查检查点

- 审查者必须确认：
  - 代码符合本宪法的所有条款。
  - Spec 已按需更新，且与实现一致。
  - 测试覆盖了变更的核心路径。
- 若发现违反宪法，审查者应标记“Request Changes”并引用具体条款。

## 六、可观测性与可调试性

### 6.1 结构化日志

- 所有外部请求（HTTP、RPC、消息队列）和关键业务操作（订单创建、支付、用户登录）必须打印结构化日志。
- 日志必须包含：`requestId`（全链路追踪 ID）、`userId`（如有）、`durationMs`（耗时）、`result`（成功/失败）。
- 使用统一的日志库（例如 `pino` 或 `winston`），禁止直接使用 `console.log`。

### 6.2 错误处理规范

- 禁止使用空 `catch` 块吞掉异常。
- 必须使用自定义错误类（继承 `Error`），并附加错误码和上下文信息（例如 `{ code: 'VALIDATION_ERROR', field: 'email' }`）。
- 在边界层（如 API 路由）统一捕获并转换为标准错误响应。

---

## 七、例外管理

### 7.1 存量代码豁免

- 对于在本宪法生效日期之前已存在的代码，如果无法立即满足某条宪法条款，可以在 `exceptions.json` 中登记豁免。
- 豁免条目必须包含：文件路径、违反的条款号、技术债 Ticket ID、计划修复日期（不得超过 6 个月）。

### 7.2 紧急情况特批

- 生产环境紧急修复（Hotfix）可临时绕过部分宪法条款，但必须在 PR 描述中注明“Hotfix 特批”，并在事后 3 个工作日内补充合规修改。

### 7.3 宪法修订

- 任何对宪法条款的增加、删除、修改，必须：
  1. 由至少两位项目技术负责人同意。
  2. 在团队 Wiki 中公示 1 周。
  3. 更新宪法版本号和生效日期。

## 八、附则

- 本宪法解释权归项目技术委员会所有。
- 违反宪法的代码不应合并到主分支。如果因特殊情况必须合并，需获得技术负责人书面批准（在 PR 中评论 `/override-constitution` 并说明理由）。
- 本宪法每半年评审一次，根据项目演进而调整。

**最后更新**：2026-04-09  
**下次评审**：2026-10-09
```

## 五、使用建议

1. **立即行动**：将上述示例中的项目名、日期、技术债格式等替换为真实内容，提交到 `.claude/rules/constitution.md`。
2. **团队宣讲**：组织一次短会（30 分钟），逐条解释宪法意图，解答疑问，确保大家理解而非抵触。
3. **工具集成**：
   - 在 CI 中添加对宪法部分条款的自动检查（如 `tsc`、`eslint`、测试覆盖率）。
   - 配置 GitHub Actions 或 GitLab CI 扫描 `exceptions.json`，对过期的豁免发出警告。
4. **逐步收紧**：每季度清理一批例外，提升标准。不要期望一步到位。
5. **AI 对齐**：在 `CLAUDE.md` 中引用宪法，并明确指示 Claude Code：“你是一个严格遵循项目宪法的 AI 助手。”

宪法不是一纸空文，而是团队共同承诺的质量底线。有了它，AI 和人类开发者才能在同一个规则体系下高效协作。
