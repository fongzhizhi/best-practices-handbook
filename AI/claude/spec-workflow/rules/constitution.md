# 项目宪法

> 版本：v1.0  
> 生效日期：2026-04-09  
> 适用范围：本项目所有源代码、配置文件、文档  
> 修订流程：需经团队技术负责人批准，并在 PR 中附上修订说明

本宪法是最高行为准则。除非明确声明例外，所有新增代码、修改代码、AI 生成代码、人工审查均必须遵守以下条款。

---

## 一、代码质量与一致性

### 1.1 TypeScript strict 模式

- **所有新编写或修改的代码**必须在 `tsconfig.json` 中开启 `strict: true` 的前提下通过编译。
- **存量代码**中的 `any` 必须标注 `// @ts-expect-error`，并附带原因及计划删除日期（格式：`// @ts-expect-error: 技术债 #1234, 计划 2026-06-01 前修复`）。
- **禁止**在新增代码中使用 `any`（包括函数参数、返回值、变量）。

### 1.2 命名规范

- 文件名：`kebab-case.ts`
- 类名、接口名、类型别名：`PascalCase`
- 变量名、函数名、参数名：`camelCase`
- 常量（`const` 声明且值不可变）：`UPPER_SNAKE_CASE`
- 私有成员：不加下划线前缀，依靠 TypeScript `private` 关键字

### 1.3 重复逻辑禁止

- 相同的业务规则或算法不得在两处及以上实现。
- AI 或人工发现重复后，必须抽取为共享函数或工具类，并替换所有调用点。

---

## 二、架构与边界清晰

### 2.1 模块边界

- 每个子仓库必须在根目录 `CLAUDE.md` 中声明其**对外暴露的公共 API**（导出的函数、类、类型）。
- **禁止**跨仓库直接导入其他仓库的 `src` 内部文件。必须通过公共 API 或共享包（`@myorg/shared-*`）访问。

### 2.2 依赖方向

- 依赖图不得出现循环（A → B → A）。
- 新增依赖前，必须运行 `npx madge --circular src/` 检查是否会引入循环。
- 依赖方向必须从具体到抽象（例如：业务模块 → 基础设施模块）。

---

## 三、文档与知识可获取

### 3.1 Spec 关联

- 任何影响**核心业务逻辑**的变更（新功能、重大 Bug 修复、重构）必须附带或更新 Spec 文档。
- Spec 文档存放于模块根目录的 `.specs/` 下，使用 Markdown 格式，至少包含：
  - 用户故事或问题描述
  - 验收场景（Given-When-Then）
  - 对现有行为的影响说明

### 3.2 CLAUDE.md 存在性

- 每个仓库根目录必须有一份 `CLAUDE.md`，描述：
  - 仓库职责（一句话）
  - 主要目录结构及作用
  - 常用开发命令（`npm run dev`、`npm test` 等）
  - 对外依赖的其他仓库或服务

---

## 四、可测试性与自动化验证

### 4.1 测试要求

- **所有新功能**必须包含至少一个单元测试（或集成测试，视模块性质而定）。
- **修复 Bug** 必须附带一个回归测试，该测试在修复前会失败、修复后通过。
- 测试覆盖率不得下降（以 `main` 分支为基准）。新增代码的行覆盖率应 ≥ 80%。

### 4.2 CI 门禁

- PR 合并前必须通过以下检查：
  - `tsc --noEmit`（TypeScript 类型检查）
  - `eslint`（使用项目配置）
  - `jest` 或对应测试框架（全部通过）
  - 构建（`npm run build`）成功
- 门禁失败时，PR 禁止合并，除非获得技术负责人特批。

---

## 五、协作与流程高效

### 5.1 PR 描述模板

- 每个 PR 的描述必须包含以下小节：

  ```markdown
  ## 变更摘要
  （简述改动内容）
  
  ## Spec 关联
  - [ ] 已更新/创建 Spec 文档（路径：`...`）
  - [ ] 不涉及核心逻辑，无需 Spec
  
  ## 测试情况
  - [ ] 单元测试通过
  - [ ] 手动测试（描述场景）
  
  ## 对核心模块的影响
  - [ ] 无影响
  - [ ] 有影响，已在 Spec 中说明
  
  ### 5.2 审查检查点

- 审查者必须确认：
  - 代码符合本宪法的所有条款。
  - Spec 已按需更新，且与实现一致。
  - 测试覆盖了变更的核心路径。
- 若发现违反宪法，审查者应标记“Request Changes”并引用具体条款。

## 六、可观测性与可调试性

### 6.1 结构化日志

- 所有外部请求（HTTP、RPC、消息队列）和关键业务操作（订单创建、支付、用户登录）必须打印结构化日志。
- 日志必须包含：`requestId`（全链路追踪 ID）、`userId`（如有）、`durationMs`（耗时）、`result`（成功/失败）。
- 使用统一的日志库（例如 `pino` 或 `winston`），禁止直接使用 `console.log`。

### 6.2 错误处理规范

- 禁止使用空 `catch` 块吞掉异常。
- 必须使用自定义错误类（继承 `Error`），并附加错误码和上下文信息（例如 `{ code: 'VALIDATION_ERROR', field: 'email' }`）。
- 在边界层（如 API 路由）统一捕获并转换为标准错误响应。

---

## 七、例外管理

### 7.1 存量代码豁免

- 对于在本宪法生效日期之前已存在的代码，如果无法立即满足某条宪法条款，可以在 `exceptions.json` 中登记豁免。
- 豁免条目必须包含：文件路径、违反的条款号、技术债 Ticket ID、计划修复日期（不得超过 6 个月）。

### 7.2 紧急情况特批

- 生产环境紧急修复（Hotfix）可临时绕过部分宪法条款，但必须在 PR 描述中注明“Hotfix 特批”，并在事后 3 个工作日内补充合规修改。

### 7.3 宪法修订

- 任何对宪法条款的增加、删除、修改，必须：
  1. 由至少两位项目技术负责人同意。
  2. 在团队 Wiki 中公示 1 周。
  3. 更新宪法版本号和生效日期。

---

## 八、附则

- 本宪法解释权归项目技术委员会所有。
- 违反宪法的代码不应合并到主分支。如果因特殊情况必须合并，需获得技术负责人书面批准（在 PR 中评论 `/override-constitution` 并说明理由）。
- 本宪法每半年评审一次，根据项目演进而调整。
