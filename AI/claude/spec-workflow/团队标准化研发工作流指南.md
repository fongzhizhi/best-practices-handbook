# 团队标准化研发工作流指南

## 一、背景与目标

本工作流基于《大型项目的渐进式高质量改造：基于 Claude Code 的实践方法论》设计，旨在通过 **AI 辅助 + 规范驱动** 的方式，将代码提交、分支管理、需求追溯、文档同步等环节全面自动化、规范化，降低开发者的心智负担，提升代码质量与协作效率。

### 核心理念

- **AI 是执行者，人是决策者**：重复性、规范性的操作交给 AI，开发者聚焦于架构设计、业务逻辑与代码审查。
- **规范即代码**：将团队约定（分支命名、提交格式、ONES 关联）固化为可执行的指令和规则，让 AI 自动遵守。
- **渐进式落地**：开发者可自由选择使用单个指令或完整工作流，不强制改变现有习惯。

## 二、整体架构速览

整个工作流由三层组成：

| 层级             | 组成部分                         | 作用                                                     |
| :--------------- | :------------------------------- | :------------------------------------------------------- |
| **智能引导层**   | `workflow-router.md` 规则        | 自动识别任务类型（新功能/Bug/重构等），主动推荐流程。    |
| **开发规范层**   | OpenSpec + Spec Miner            | 规范驱动开发，确保需求、设计、代码、文档同步。           |
| **自动化指令层** | `/git-*`、`/pre-commit` 系列指令 | 将 Git 操作、格式化、提交、MR 创建等环节封装为智能指令。 |

**一句话理解**：AI 先告诉你该走哪条路（引导层），然后陪你按标准把路走完（规范层+指令层）。

## 三、快速上手：五分钟完成一次提交

如果你是第一次接触，按以下步骤体验完整流程：

```bash
# 1. 切换到开发基准分支（如 develop）
/git-checkout develop

# 2. 创建功能分支（AI 会智能生成分支名）
/git-branch

# 3. 进行代码开发...

# 4. 开发完成，一键推送并创建 MR（自动格式化、生成规范提交）
/git-push

# 5. 创建合并请求
/git-merge
```

以上五步即可完成从分支创建到 MR 提交的全过程。接下来详细介绍每个指令的用法。

## 四、核心指令详解

所有指令均以 `/` 开头，在 Claude Code 对话中直接输入即可。

### 4.1 分支与切换

| 指令                   | 用途                                                       | 示例                                    |
| :--------------------- | :--------------------------------------------------------- | :-------------------------------------- |
| `/git-checkout <分支>` | 切换分支并拉取最新代码。支持远程分支自动创建本地跟踪分支。 | `/git-checkout main`                    |
| `/git-branch`          | 智能创建符合命名规范的分支。自动处理未提交变更、冲突。     | `/git-branch` 或 `/git-branch feat/xxx` |
| `/git-stash`           | 暂存当前未提交变更，生成有意义的消息。                     | `/git-stash`                            |

### 4.2 代码质量与提交

| 指令               | 用途                                                         | 示例               |
| :----------------- | :----------------------------------------------------------- | :----------------- |
| `/prettier-format` | 对变更文件执行 Prettier 格式化（自动检测项目配置）。         | `/prettier-format` |
| `/pre-commit`      | 提交前质量检查：类型安全、命名、复杂度、测试覆盖、Spec 同步等。 | `/pre-commit`      |
| `/git-commit`      | 生成符合 Conventional Commits 规范的提交信息，自动关联 ONES 单和 Spec。 | `/git-commit`      |

### 4.3 推送与合并

| 指令         | 用途                                                         | 示例                                            |
| :----------- | :----------------------------------------------------------- | :---------------------------------------------- |
| `/git-push`  | 推送当前分支，自动处理未提交代码、主分支保护、冲突等。       | `/git-push`                                     |
| `/git-merge` | 通过 Git Push Options 创建 GitLab MR（无需额外 CLI）。自动填充目标分支和审核人。 | `/git-merge --target develop --reviewer @alice` |

## 五、规范驱动开发：OpenSpec 与 Spec Miner

### 5.1 核心概念

- **OpenSpec**：先写规范，再写代码。适用于新功能、有计划的重构。
- **Spec Miner**：从旧代码逆向提取规范。适用于理解无文档的遗留模块。

### 5.2 何时使用哪个？

| 场景         | 使用工具              | 操作                                             |
| :----------- | :-------------------- | :----------------------------------------------- |
| 新功能开发   | OpenSpec              | `/openspec:proposal` → `/openspec:apply`         |
| 修复 Bug     | Spec Miner → OpenSpec | 先逆向分析模块，再创建修复提案                   |
| 重构遗留代码 | Spec Miner → OpenSpec | 先提取现有行为作为安全基线，再创建重构提案       |
| 理解旧代码   | Spec Miner            | 直接对话：“用 spec-miner-openspec 分析 xxx 模块” |

### 5.3 常用命令

| 命令                 | 说明                                  |
| :------------------- | :------------------------------------ |
| `/openspec:proposal` | 创建变更提案（proposal.md、tasks.md） |
| `/openspec:apply`    | 执行已批准的变更提案                  |
| `/openspec:archive`  | 归档已完成的变更，更新主规范库        |

Spec Miner 已封装为 `spec-miner-openspec` 技能，使用自然语言触发即可，例如：

> “用 spec-miner-openspec 分析 src/legacy/order.ts，模块名设为 order”

生成的规范文档会自动存入 `openspec/specs/` 目录。

## 六、典型场景完整示例

### 场景一：开发新功能（用户头像上传）

1. **准备分支**  
   `/git-checkout develop` → `/git-branch`（AI 生成 `main/feat/PRO-12345`）

2. **编写规范**  
   `/openspec:proposal 实现头像上传功能，支持 JPG/PNG，大小不超过 5MB...`

3. **实施编码**  
   `/openspec:apply avatar-upload`，在 AI 辅助下完成代码。

4. **提交合并**  
   `/pre-commit` → `/git-commit` → `/git-push` → `/git-merge -t develop`

5. **归档规范**  
   `/openspec:archive avatar-upload`

### 场景二：修复 Bug（登录无响应）

1. **定位问题**  
   “用 spec-miner-openspec 分析 src/auth/login.ts，模块名 auth”

2. **创建修复提案**  
   `/openspec:proposal 修复登录失败时 UI 无反馈`（AI 自动关联 ONES 单）

3. **实施并提交**  
   `/openspec:apply fix-login-feedback` → `/git-push` → `/git-merge`

### 场景三：重构历史遗留模块

1. **提取现状规范**  
   “用 spec-miner-openspec 分析 src/legacy/helper.ts，模块名 legacy/helper”

2. **设计重构方案**  
   `/openspec:proposal 拆分 Helper.ts 为三个独立模块`

3. **逐步实施**  
   `/openspec:apply refactor-helper`，每步运行测试。

4. **提交归档**  
   `/git-push` → `/git-merge` → `/openspec:archive`

## 七、智能引导：让 AI 主动帮你

无需记忆所有指令，`workflow-router.md` 规则会让 Claude Code 自动识别你的意图并引导你。

**示例对话**：

> **你**：订单金额计算不对，用了优惠券后多扣了10块。  
> **AI**：我识别到这是一个 **Bug 修复**任务。建议先用 Spec Miner 分析相关模块，再创建修复提案。需要我帮你分析吗？

你只需描述问题或意图，AI 会推荐合适的工作流并主动调用对应指令。

## 八、配置与安装

### 8.1 前置要求

- Claude Code CLI 已安装并登录
- Git ≥ 2.18
- GitLab 版本 ≥ 11.10（使用 Push Options 创建 MR 所需）
- Node.js ≥ 20.19.0（用于 OpenSpec）

### 8.2 安装 OpenSpec 技能包

```bash
npm install -g @fission-ai/openspec@latest
cd your-project
openspec init  # 选择 Claude Code
```

### 8.3 安装 Spec Miner 技能包

网上推荐的方案较为繁琐，也不适合离线环境安装，更简单有效的方案，网上检索并下载 Spec Miner 技能包（一些 Markdown 文件），复制到`.claude/skills/` 即可。

### 8.4 部署工作流文件

另外将以下自定义指令和技能包放置到项目对应目录：

```txt
项目根目录/
├── .claude/
│   ├── commands/				# 自定义的快捷指令
│   │   ├── git-checkout.md
│   │   ├── git-branch.md
│   │   ├── git-stash.md
│   │   ├── prettier-format.md
│   │   ├── git-commit.md
│   │   ├── git-push.md
│   │   ├── git-merge.md
│   │   └── pre-commit.md
│   ├── skills/					# 自定义的技能包
│   │   └── spec-miner-openspec/
│   │       └── SKILL.md
│   └── rules/
│       ├── constitution.md      # 编码规范
│       └── workflow-router.md   # 智能引导规则
└── CLAUDE.md                    # 引用上述规则
```

在 `CLAUDE.md` 中添加：

```markdown
## 项目配置

请严格遵守以下规则：

### 编码规范

@.claude/rules/constitution.md

### 工作流引导

@.claude/rules/workflow-router.md
```

## 九、常见问题

**Q：我习惯先写代码再创建分支，会影响使用吗？**  
A：不会。`/git-branch` 会智能检测未提交代码，自动通过 stash 迁移到新分支。

**Q：如果 ONES 单号没有在对话中提及，`/git-commit` 还能自动关联吗？**  
A：不能自动提取，但你可以手动指定：`/git-commit --ones PRO-151081`。

**Q：我不用 GitLab，能用这套工作流吗？**  
A：`/git-merge` 基于 GitLab Push Options 设计，若使用 GitHub，可稍作适配或使用 `gh` CLI。其余指令均为通用 Git 操作。

**Q：指令太多记不住怎么办？**  
A：无需记忆，智能引导规则会自动推荐。你只需描述任务，AI 会提示下一步该用什么指令。

**Q：我不想用 OpenSpec，可以跳过吗？**  
A：可以。AI 引导时会提供跳过选项，但建议至少为重要变更创建提案，以便追溯。
