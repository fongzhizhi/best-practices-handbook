请对当前分支的变更执行提交前质量检查，生成一份结构化的检查单。本指令帮助在提交或推送前发现常见问题，减少 MR 中的审查往返。

## 执行步骤

### 1. 获取变更范围
- 执行 `git diff --name-only` 获取工作区变更文件。
- 执行 `git diff --cached --name-only` 获取暂存区变更文件。
- 合并去重，分类为：
  - **source**：业务代码（`.ts`、`.tsx`、`.js`、`.jsx`）
  - **spec**：规格文档（`openspec/specs/**/*.md`、`spec.md`）
  - **test**：测试文件（`*.test.ts`、`*.spec.ts`）
  - **config**：配置文件（`package.json`、`tsconfig.json`、`.eslintrc` 等）

### 2. 类型安全检查
对 `source` 文件执行以下检查：
- **新增 `any` 类型**：扫描变更行中是否引入 `any`（排除项目 `.claude/rules/constitution.md` 中声明的例外场景）。
- **缺失返回类型**：检查导出的函数或方法是否缺少显式返回类型注解。
- **隐式类型推断风险**：标注依赖 `strict` 模式可能隐式推导为 `any` 的变量。

### 3. 命名与风格检查
- **命名规范**：检查变量、函数、类、文件名是否符合团队约定（默认读取 `.claude/rules/constitution.md` 中的命名章节）。
- **魔法数字**：标记直接出现在代码中的数字字面量，建议提取为命名常量。

### 4. 复杂度与重复检查
- **圈复杂度**：对新增或修改的函数计算圈复杂度，超过阈值（默认 10）时输出警告。
- **重复代码**：与项目内现有代码比对，若存在 3 处以上高度相似的代码块，建议抽取公共函数。

### 5. 错误处理检查
- **空 `catch` 块**：标记所有空的 `catch (e) {}` 或仅包含注释的异常捕获。
- **错误吞没**：检查 `catch` 块中是否缺少错误上报或用户提示。

### 6. 测试覆盖检查
- 对于 `source` 文件中新增或修改的导出函数，检查是否存在对应的测试文件。
- 若无对应测试，且不在项目豁免清单中，标记为**缺失测试**。

### 7. Spec 一致性检查
- 对于 `source` 文件所在的模块，尝试在同目录或 `openspec/specs/` 下定位对应的 `spec.md`。
- 若存在 Spec，检查本次变更是否：
  - 修改了公开接口但未更新 Spec。
  - 新增了功能但未在 Spec 中记录验收场景。
  - 修复了 Bug 但未在 Spec 中补充回归用例。
- 若涉及修改但 Spec 未更新，标记为 **Spec 漂移**。

### 8. 生成检查单
汇总以上检查结果，按以下格式输出：

```markdown
## 🔍 Pre-Commit 检查单

**分支**：<当前分支名>
**变更文件数**：X 个（source: X, spec: X, test: X）

### ✅ 已通过项
- [x] 无新增 ESLint 错误
- [x] 文件命名符合规范
- [x] 配置文件变更合理

### ❌ 需处理项（阻断提交）

| 文件                  | 问题类型  | 详情                                        | 建议操作                       |
| :-------------------- | :-------- | :------------------------------------------ | :----------------------------- |
| `src/user/avatar.ts`  | 类型安全  | 新增 2 处 `any` 类型                        | 定义具体接口替换               |
| `src/user/profile.ts` | 复杂度    | `formatDisplayName` 圈复杂度 12             | 拆分为多个小函数               |
| `src/user/avatar.ts`  | 测试缺失  | 新增导出函数 `resizeAvatar` 无对应测试      | 在 `avatar.test.ts` 中补充用例 |
| `specs/user/spec.md`  | Spec 漂移 | 代码新增 `getAvatarUrl` 导出，Spec 中未记录 | 更新 Spec 的 API 章节          |

### ⚠️ 建议关注项（非阻断）

| 文件                 | 问题类型 | 详情                                                         | 建议操作               |
| :------------------- | :------- | :----------------------------------------------------------- | :--------------------- |
| `src/user/avatar.ts` | 重复逻辑 | `getExtension` 与 `utils/file.ts` 中的 `extractExt` 相似度 85% | 考虑复用或抽取公共函数 |

```


### 9. 退出码与后续提示
- 若存在 ❌ 项，指令以非零状态退出，并提示：

  ```txt
  ❌ 检查未通过，请处理以上阻断项后重新运行 /pre-commit。
- 若仅有 ⚠️ 项或无问题，提示：
  ```txt
  ✅ 检查通过，可以执行 /git-commit 或 /git-push 提交代码。
  ```

## 使用示例
```bash
/pre-commit                     # 对当前所有变更执行检查
/pre-commit --strict            # 严格模式，将建议关注项也视为阻断
```

## 配置选项
可在 `.claude/config.json` 中自定义阈值与豁免规则：
```json
{
  "preCommit": {
    "complexityThreshold": 10,
    "allowAnyIn": ["*.test.ts", "*.stories.tsx"],
    "ignorePatterns": ["dist/**", "coverage/**"]
  }
}
```

## 注意事项
- 本指令仅作静态分析，不替代 CI 门禁中的完整检查（如 `tsc`、`eslint`、测试运行）。
- Spec 一致性检查依赖 Spec 文档的规范存放位置（`openspec/specs/`），请确保项目遵循此约定。
