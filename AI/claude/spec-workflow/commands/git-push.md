请将当前分支的提交推送到远程仓库，并自动处理代码格式化、提交信息生成等前置步骤。本指令可独立使用，智能应对主分支保护、无变更、上游缺失、远程冲突等场景。

## 执行步骤

### 1. 状态检测
- 执行 `git status --porcelain` 检查工作区和暂存区状态。
- 执行 `git branch --show-current` 获取当前分支名。
- 执行 `git rev-parse --abbrev-ref @{upstream} 2>/dev/null` 检查上游跟踪分支。
- 执行 `git diff --quiet origin/<当前分支名>..HEAD 2>/dev/null` 检查是否有未推送的本地提交（若远程分支存在）。

### 2. 前置检查：无变更处理
- 若工作区干净（无未提交变更）且无未推送提交，输出：

  ```txt
  ℹ️ 没有需要推送的提交，工作区已是最新状态。
  ```

  流程终止。

### 3. 前置检查：未提交变更处理
- 若存在未提交变更（工作区或暂存区非空），提示用户：

  ```txt
  ⚠️ 当前有未提交的变更，推送前需先提交。
  请选择处理方式：
  A. 自动格式化并提交（调用 /prettier-format 和 /git-commit）
  B. 暂存变更并继续推送（调用 /git-stash）
  C. 取消操作，手动处理
  ```

    - 若选 `A`：依次调用 `/prettier-format` 和 `/git-commit`，完成后继续后续步骤。

    - 若选 `B`：调用 `/git-stash` 暂存变更，完成后继续后续步骤。

    - 若选 `C`：终止流程。


### 4. 主分支保护检查
- 若当前分支名为 `main`、`master`、`develop` 、`dev`等受保护分支，推送前需确认： 

  ```txt
  ⚠️ 当前分支 `<当前分支名>` 可能为受保护分支，不建议直接推送。
  请选择处理方式：
  A. 创建功能分支并迁移变更（推荐）
  B. 强制推送（需二次确认）
  C. 取消操作
  ```

  - 若选 `A`：
    1. 调用 `/git-branch` 创建新功能分支（场景 A，迁移未提交变更）。
    2. 使用新分支名继续后续推送流程。
  - 若选 `B`：
    1. 二次确认：`⚠️ 即将强制推送到受保护分支，该操作可能影响他人。确认继续？(y/n)`
    2. 确认后继续后续步骤，推送时将使用 `--force-with-lease`（见步骤 7）。
  - 若选 `C`：终止流程。

### 5. 远程分支存在性检查与上游设置
- 执行 `git fetch origin` 更新远程分支列表。
- 执行 `git show-ref --verify --quiet refs/remotes/origin/<当前分支名>` 检查远程是否存在同名分支。

#### 5.1 远程分支不存在
- 提示用户：

  ```txt
  ℹ️ 远程分支 origin/<当前分支名> 不存在，将创建并设置上游。
  ```

- 推送时自动附加 `-u` 参数：`git push -u origin <当前分支名>`。

#### 5.2 远程分支已存在但无本地跟踪
- 若上游未设置但远程分支存在，自动设置跟踪：

  ```bash
   git branch --set-upstream-to=origin/<当前分支名> <当前分支名>
  ```

- 然后执行普通推送。

### 6. 推送前冲突预检
- 检查本地与远程分支是否分叉（执行 `git rev-list --count origin/<当前分支名>..HEAD` 和反向计数）。

- 若远程有新提交且本地未合并，提示用户：

  ```txt
  ⚠️ 远程分支 origin/<当前分支名> 有新的提交，推送可能被拒绝。
  请选择处理方式：
  A. 变基后推送（git pull --rebase）
  B. 合并后推送（git pull --merge）
  C. 强制推送（需二次确认，覆盖远程历史）
  D. 取消操作
  ```

    - 若选 `A`：执行 `git pull --rebase origin <当前分支名>`，解决可能冲突后推送。

    - 若选 `B`：执行 `git pull --no-rebase origin <当前分支名>`，解决可能冲突后推送。

    - 若选 `C`：二次确认后使用 `git push --force-with-lease`。

    - 若选 `D`：终止流程。


### 7. 执行推送
- 根据上述步骤确定参数，执行最终推送命令。

- 若用户曾选择强制推送，附加 `--force-with-lease`。

- 输出推送结果：

  ```txt
  ✅ 推送成功：<当前分支名> → origin/<当前分支名>
  ```

### 8. 推送后操作提示
- 若之前暂存了变更（步骤 3 选 B），询问是否恢复：

  ```txt
  是否恢复之前暂存的变更？(y/n)
  ```

  - 若选 `y`：执行 `git stash pop`，若有冲突提示手动解决。

- 提示后续操作：

  ```txt
  💡 下一步可执行 /git-merge 创建合并请求。  

## 使用示例
```bash
# 标准推送（自动处理格式化和提交）
/git-push

# 跳过前置格式化步骤
/git-push --skip-format

# 强制推送（需二次确认）
/git-push --force
```

## 注意事项
- 本指令会调用 `/prettier-format`、`/git-commit`、`/git-stash`、`/git-branch`，请确保对应指令文件存在。
- 强制推送前请确认不会覆盖他人工作，`--force-with-lease` 比 `--force` 更安全。
- 若推送过程中因权限问题失败，请检查 GitLab 账号认证状态（`glab auth status`）。
