请根据当前工作状态智能创建符合团队规范的 Git 分支。本指令可独立使用，能处理代码已完成和代码未开始两种场景，自动处理未提交变更、智能生成分支名并解决冲突。

## 执行步骤

### 1. 状态检测
- 执行 `git status --porcelain` 检查是否有未提交的变更。
- 执行 `git branch --show-current` 获取当前分支名。

根据检测结果进入对应场景。

### 2. 场景 A：代码已完成（存在未提交变更）

此场景下，当前分支通常是开发主分支（如 `main`、`develop`）或临时工作分支，需为本次提交创建专属功能分支。

#### 2.1 检查当前分支
- 若当前分支为 `main`、`master`、`develop` 等受保护分支，则必须创建新分支。

- 若当前分支已是功能分支（如 `feat/xxx`、`fix/xxx`），询问用户：

  ```txt
  当前已在分支 `<当前分支名>` 上。是否直接使用该分支提交？(y/n)
  ```

    - 若选 `y`，则流程结束，后续直接使用 `/git-commit` 提交。

    - 若选 `n`，则继续创建新分支。


#### 2.2 智能生成分支名

扫描当前对话历史，识别 ONES 单的特征：
- **链接格式**：包含 `ones.` 的 URL
- **ID 格式**：`[A-Z]+-\d+`，如 `PRO-151081`
- **标题格式**：紧跟在 ID 后的中文描述，如 `【编辑器】编辑器打开PCB报错，提示无权限`

若存在多个 ONES 单，选择最近一次提及或与变更最相关的。

**若识别到 ONES 单**，按以下规则生成分支名：

```txt
<分支来源>/<变更类型>/<ONES_ID>
```

- **分支来源**：提取当前分支名中最后一个 `/` 之后的部分作为来源标识。例如当前分支为 `origin/main`、`upstream/develop` 或 `release/3.2`，则分别取 `main`、`develop`、`3.2`。
- **变更类型**：根据变更内容判断，取值 `feat` / `fix` / `refactor` / `docs` / `style` / `test` / `chore` / `perf` / `ci`。
- **ONES_ID**：扫描到的 ONES 单 ID。

示例：当前分支为 `origin/main`，变更类型为 `fix`，ONES ID 为 `PRO-151081`，则生成：

```txt
main/fix/PRO-151081
```
生成后询问用户：

```txt
建议分支名：main/fix/PRO-151081
是否使用此分支名？(y/n/手动输入)
```

- 若选 `y`，使用该名称。
- 若选 `n` 或选择手动输入，则接收用户输入的分支名。

**若未提取到 ONES 信息**，直接询问用户输入分支名：

```markdown
请输入新分支名，推荐格式：`<分支来源>/<变更类型>/<ONES_ID或描述>`

格式说明：

- **分支来源**：当前基准分支名或版本标识，如 `main`、`develop`、`3.2`
- **变更类型**：本次提交的性质，如 `feat` / `fix` / `refactor` / `docs` / `style` / `test` / `chore` / `perf` / `ci`
- **ONES_ID或描述**：关联的 ONES 单号，若无则用简短英文描述（如 `fix-history-back`）

示例：

- `main/feat/PRO-151081`
- `develop/fix/修复历史记录回退失败问题`
- `3.2/refactor/优化权限校验逻辑`

请输入分支名：
```

#### 2.3 分支名冲突检查与处理

确定分支名后，执行 `git branch --list <分支名>` 检查本地是否已存在同名分支。

##### 若分支名不存在
直接进入步骤 2.4。

##### 若分支名已存在
输出以下提示，等待用户选择：

```txt
⚠️ 已存在同名分支 `<分支名>`，请选择处理方式：
A. 缓存当前变更，切换到已存在分支
B. 删除已存在旧分支，新建同名分支
C. 手动处理，退出流程
```

**选项 A：缓存变更并切换**
1. 调用 `/git-stash` 暂存当前变更（若指令存在则直接调用；否则分析变更内容生成 stash 消息并执行 `git stash push -m "<消息>"`）。
2. 切换到已存在分支：`git checkout <分支名>`。
3. 询问用户是否恢复暂存的变更：
   - 若确认，执行 `git stash pop`。若有冲突，提示用户手动解决。
4. 流程结束，输出当前分支状态。

**选项 B：删除旧分支并新建**
1. 二次确认：

   ```txt
   ⚠️ 即将强制删除本地分支 `<分支名>`，该操作不可逆。确认继续？(y/n)
   ```

2. 若用户确认：
   - 执行 `git branch -D <分支名>`。
   - 继续步骤 2.4。

3. 若取消，返回分支名冲突选择。

**选项 C：手动处理**
- 输出提示：`流程已终止，请手动处理分支冲突后重新执行 /git-branch。`
- 退出指令。

#### 2.4 暂存当前变更
为安全创建新分支，需先将未提交变更暂存：
- 若 `/git-stash` 指令存在，则调用它执行暂存（可传递自定义消息）。
- 否则，分析变更内容生成有意义的 stash 消息，执行 `git stash push -u -m "<缓存信息>"`。

#### 2.5 确定基准分支并创建新分支
- 若用户指定 `--base <分支名>`，则以该分支为基准。
- 若未指定，则默认以**当前分支**为基准。
- 执行 `git checkout -b <新分支名> <基准分支>`。

#### 2.6 恢复暂存变更
- 执行 `git stash pop` 将步骤 2.4 暂存的变更恢复到新分支。
- 若有冲突，提示用户手动解决，并给出冲突文件列表。

#### 2.7 输出结果

```txt
✅ 已创建并切换到新分支：<分支名>
✅ 基准分支：<基准分支>
✅ 未提交变更已迁移至新分支。
```

### 3. 场景 B：代码未开始（工作区干净）

此场景下，用户准备开始新工作，需创建新分支。

#### 3.1 解析用户意图
从用户输入或上下文中提取：
- 是否提供了 ONES 单信息？
- 是否指定了分支类型？（feat / fix / refactor 等）

#### 3.2 生成或询问分支名
- 若有 ONES 单信息，按场景 A 的规则生成建议分支名，并询问用户确认（y/n/手动输入）。
- 若无足够信息，直接询问用户输入分支名（提示推荐格式同场景 A）。

#### 3.3 确定基准分支并创建
- 若用户指定 `--base <分支名>`，则以该分支为基准。
- 若未指定，则以**当前分支**为基准。
- 执行 `git checkout -b <新分支名> <基准分支>`。

#### 3.4 输出结果

```txt
✅ 已创建并切换到新分支：<分支名>
✅ 基准分支：<基准分支>
现在可以开始开发工作了。
```

## 使用示例
```bash
# 场景A：代码已改完，智能创建分支
/git-branch

# 场景A：指定基准分支
/git-branch --base origin/main

# 场景B：开始新功能，从 develop 创建
/git-branch --base develop

# 场景B：指定分支名创建
/git-branch feat/avatar-upload --base main
```

## 注意事项
- 分支名中的空格和特殊字符会自动替换为 `-`。
- 若 `git stash pop` 产生冲突，请手动解决后继续。
- 本指令依赖 `/git-stash` 提供更智能的暂存体验，若无此指令，会自动降级为原生 `git stash` 命令。
