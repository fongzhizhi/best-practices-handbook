# 工作流优化指南：从 Commands/Skills 到 Subagents 的架构升级

## 一、当前工作流回顾与问题诊断

我们已构建了一套基于 Claude Code 的完整研发工作流，包含三个核心层次：

| 层级       | 组成部分                                        | 当前职责                                   |
| :--------- | :---------------------------------------------- | :----------------------------------------- |
| **决策层** | `CLAUDE.md` + `rules/`                          | 项目架构、编码规范、场景识别与路由         |
| **执行层** | `/git-*` Commands + `spec-miner-openspec` Skill | Git 自动化、代码格式化、质量检查、规范提取 |
| **配置层** | `settings.json`                                 | 权限管理、GitLab Token 配置                |

这套工作流已实现从分支创建到 MR 合并的完整闭环，但在实际使用中暴露出两个核心问题：

1. **执行速度慢**：每个 Command 都包含了大量场景判断逻辑（如检测未提交代码、处理冲突选项、智能生成分支名），导致 AI 需要多轮“思考-执行”，单次操作耗时数分钟。
2. **职责臃肿**：Commands 和 Skills 既是“决策者”又是“执行者”，既做状态检测又做原子操作，维护成本高且复用性差。

**根本原因**：我们将“流程智能”错误地放在了执行层，导致每个指令都试图成为“全能管家”。


## 二、为什么需要升级到 Subagents 模式？

Subagents（子代理）是 Claude Code 提供的“角色工程化”能力，允许我们创建具有独立上下文、专属工具和特定职责的 AI 代理。其核心优势在于：拥有独立的上下文窗口，可并行执行多个子任务，实现调度与执行的天然分离，并按 Agent 配置工具白名单来保障安全。

**核心理念转变**：从“让每个指令都智能”转变为“让调度层智能，让执行层纯粹”。


## 三、目标架构：三层分离模型

我们将构建一个“调度智能 + 执行纯粹 + 能力复用”的新架构：

```
┌─────────────────────────────────────────────────────────────────┐
│              调度层（Orchestrator Agent + workflow-router）       │
│  职责：场景识别、任务拆解、流程编排、异常决策、Agent 调度          	   │
│  特点：高度智能，负责所有“判断”和“选择”                                │
└──────────────────────────────────────────────────────────────────┘
                              │ 调用
          ┌───────────────────┼───────────────────┐
          ▼                   ▼                   ▼
┌─────────────────┐ ┌─────────────────┐ ┌─────────────────┐
│  专业 Agent 层   │ │  专业 Agent 层   │ │  专业 Agent 层   │
│   (analyzer)    │ │  (spec-writer)  │ │  (implementer)  │
├─────────────────┤ ├─────────────────┤ ├─────────────────┤
│ 职责：执行特定   │ │ 职责：执行特定   │ │ 职责：执行特定   │
│ 领域的复杂任务   │ │ 领域的复杂任务   │ │ 领域的复杂任务   │
│ 特点：领域专注   │ │ 特点：领域专注   │ │ 特点：领域专注   │
└─────────────────┘ └─────────────────┘ └─────────────────┘
                              │ 复用
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│              原子能力层（瘦身后的 Commands + Skills）              │
│  职责：纯粹的单一操作，不做任何决策和交互                           │
│  特点：极简、快速、可被任意上层组件调用                            │
└─────────────────────────────────────────────────────────────────┘
```

### 各层职责明确界定

| 层级              | 可以做的事                                                 | 不可以做的事                           |
| :---------------- | :--------------------------------------------------------- | :------------------------------------- |
| **调度层**        | 分析用户意图、选择工作流、决定调用哪个 Agent、处理异常分支 | 直接执行 Git 命令、直接写文件          |
| **专业 Agent 层** | 编排领域内步骤、调用原子能力、处理领域内异常               | 跨领域决策、直接与用户进行流程性交互   |
| **原子能力层**    | 执行单一 Git 命令、执行格式化、生成 commit message         | 做状态检测、弹交互选项、做分支策略判断 |


## 四、旧有组件改造指导

### 4.1 改造总览

| 原有组件                    | 当前问题                                 | 改造方向                                                  | 在新架构中的定位      |
| :-------------------------- | :--------------------------------------- | :-------------------------------------------------------- | :-------------------- |
| `/git-checkout`             | 包含未提交检测、stash 交互、远程分支处理 | 移除所有决策逻辑，退化为纯 `git checkout` 执行器          | 原子能力层            |
| `/git-branch`               | 包含分支名生成、冲突处理、stash 迁移     | 移除智能生成和交互，退化为纯 `git checkout -b`            | 原子能力层            |
| `/git-commit`               | 包含格式化前置、暂存区检测               | 格式化前置上移到 Agent，仅保留 message 生成和 commit 执行 | 原子能力层            |
| `/git-push`                 | 包含未提交检测、主分支保护、冲突处理     | 移除所有检测和保护逻辑，退化为纯 `git push`               | 原子能力层            |
| `/git-merge`                | 包含前置状态检测、目标分支/审核人获取    | 决策逻辑上移到 Agent，仅保留 API 调用                     | 原子能力层            |
| `/prettier-format`          | 包含变更文件检测                         | 保持简单，或上移到 Agent                                  | 原子能力层            |
| `spec-miner-openspec` Skill | 包含迁移到 OpenSpec 目录的流程逻辑       | 移除流程逻辑，仅保留逆向分析的核心能力                    | 专业 Agent 的能力插件 |
| `workflow-router.md`        | 规则文件，被动引导                       | 升级为 Orchestrator Agent 的调度规则                      | 调度层的决策依据      |
| `constitution.md`           | 编码规范                                 | 保持不变，作为共享知识                                    | 所有层的共享规范      |

### 4.2 Commands 瘦身具体要做的事

**核心原则**：每个 Command 只做**一件事**，不做任何状态检测和用户交互。遇到异常直接报错退出，由上层 Agent 决定如何处理。

**具体改造清单**：

1. **删除所有 `git status --porcelain` 检测**——不再判断是否有未提交变更，不再提供 stash 或 commit 选项。
2. **删除所有交互式询问**——不再弹出 A/B/C 选项，不再询问“是否继续”。
3. **删除所有智能生成逻辑**——`/git-branch` 不再生成分支名，`/git-commit` 不再提取 ONES 信息（此能力保留但改为纯函数式，由 Agent 传入参数）。
4. **删除所有策略判断**——`/git-push` 不再判断主分支保护，`/git-checkout` 不再处理远程分支不存在的情况。
5. **保留纯粹的执行能力**——仅执行对应的 Git 命令或格式化操作。

**改造后效果**：每个 Command 文档从 100+ 行缩减至 10-20 行，执行速度从分钟级降至秒级，可被任意上层组件安全调用，无副作用。

### 4.3 Skills 瘦身具体要做的事

**核心原则**：Skills 只提供领域知识和方法论，不包含流程控制和文件迁移等编排逻辑。

**具体改造清单**：

1. **移除文件迁移逻辑**——`spec-miner-openspec` 不再负责将产物移动到 `openspec/specs/`，此职责交由调用它的 Agent 完成。
2. **移除交互式询问**——不再询问模块名、是否覆盖等，所需参数由 Agent 传入。
3. **聚焦核心能力**——仅保留“分析代码 → 输出规范摘要”的核心能力，输出格式标准化，便于 Agent 解析。

**改造后效果**：Skill 成为纯粹的“能力插件”，可被不同 Agent 在不同场景复用；Agent 负责“何时用”和“用后怎么办”，Skill 只负责“怎么做”。

### 4.4 workflow-router 升级要做的事

**当前状态**：一个被动的规则文件，AI 读取后执行引导。

**目标状态**：作为 Orchestrator Agent 的调度规则，主动进行场景识别和 Agent 调度。

**具体升级事项**：

1. **将规则文件转化为 Agent 的系统提示**——创建 `orchestrator` Agent，将其规则作为该 Agent 的行为准则。
2. **增加 Agent 调度映射表**——定义每个场景应调用哪个专业 Agent，定义 Agent 间的交接逻辑。
3. **增加异常处理策略**——当子 Agent 失败时，主控 Agent 如何响应，何时需要人工介入。
4. **保持与用户的自然交互**——主控 Agent 负责所有与用户的流程性对话，子 Agent 只通过主控 Agent 间接获取信息。

### 4.5 新增专业 Agent 要做的设计

**需要创建的 Agent 及职责**：

| Agent 名称     | 核心职责                     | 调用的原子能力                                 | 输出                  |
| :------------- | :--------------------------- | :--------------------------------------------- | :-------------------- |
| `analyzer`     | 分析遗留代码，定位 Bug 根因  | `spec-miner-openspec` Skill                    | 行为摘要、风险提示    |
| `spec-writer`  | 创建和维护 OpenSpec 规范     | `openspec` Commands、文件读写                  | proposal.md、tasks.md |
| `implementer`  | 按规范编码并完成交付         | 瘦身后的 `/git-*` Commands、`/prettier-format` | 代码变更、MR 链接     |
| `orchestrator` | 场景识别、任务调度、流程编排 | 上述三个 Agent                                 | 任务完成报告          |

**设计要点**：
- 每个 Agent 的 `description` 必须清晰描述其触发条件
- 每个 Agent 的 `tools` 白名单仅包含其职责所需的原子能力
- Agent 之间不直接通信，所有协调通过主控 Agent 完成

## 六、实施路线

### 第一阶段：原子能力抽离（1-2 天）
- 将所有 `/git-*` Commands 瘦身，移除决策逻辑
- 将 `spec-miner-openspec` Skill 中的文件迁移逻辑移除
- 验证每个原子能力可独立正常运行

### 第二阶段：专业 Agent 构建（2-3 天）
- 创建 `analyzer`、`spec-writer`、`implementer` 三个 Agent
- 为每个 Agent 配置合适的工具白名单和 Skills
- 在单个场景（如新功能开发）中测试 Agent 协作

### 第三阶段：主控 Agent 集成（1-2 天）
- 将 `workflow-router.md` 的规则转化为 `orchestrator` Agent 的系统提示
- 实现场景识别 → Agent 调度 → 结果汇总的完整链路
- 测试多场景切换（新功能、Bug 修复、重构）

### 第四阶段：验证与优化（持续）
- 对比新旧工作流的速度和成功率
- 根据反馈调整 Agent 的职责边界
- 逐步将更多场景纳入 Agent 模式

## 八、总结

本次架构升级的核心是**职责分离**：

- **改造前**：每个 Command 都是“决策者+执行者”，臃肿且低效。
- **改造后**：调度层负责“判断”，专业 Agent 层负责“编排”，原子能力层负责“执行”。

这一改造将带来三个直接收益：
1. **速度提升**：原子命令不再需要 AI 思考，执行时间从分钟级降至秒级。
2. **维护简化**：Commands 退化为极简执行器，修改成本极低。
3. **能力复用**：原子能力可被任意 Agent 组合调用，形成新的工作流。

最终，我们将从“一堆智能但笨重的指令”进化到“一套分工明确、高效协作的 Agent 团队”。


## 九、方案验证后的未来演进参考

### 9.1 当前方案在主流生态中的定位

本文档提出的“调度层 + 专业 Agent 层 + 原子能力层”三层架构，与 2025-2026 年社区最佳实践高度一致。社区中的多个标杆项目——如 awesome-claude-code（约 12.1k Star）、包含 75+ 子代理的 claude-code-subagents-collection（约 9.9k Star）——均采用了类似的 Agent 分工模式。Anthropic 官方推出的 Agent Skills 知识库（近 29k Star）更是直接印证了这一架构的设计方向。

同时，从“vibe coding”向“Spec-Driven Development (SDD)”和“Context Engineering”的范式迁移，已成为 2025-2026 年 AI 编程领域最显著的趋势。我们的方案正是这一趋势的工程化落地。

### 9.2 三个递进的演进方向

基于当前方案的验证结果，建议团队按以下优先级逐步探索更深层次的优化。

#### 方向一：深化 SDD 集成——让规范成为唯一真相源（优先级：高，短期可实现）

当前我们已使用 OpenSpec 管理规范，但规范的创建、验证和代码同步仍存在人工介入点。下一步可以：

- **引入 Spec Kit 自动化链路**：GitHub 官方开源的 Spec Kit 提供了 spec → plan → tasks → implement → review → merge 的完整工具链，可与 Claude Code 无缝集成。
- **将 Spec 验证纳入 Agent 职责**：创建专门的 `spec-validator` Agent，在每次代码变更后自动检查 Spec 与代码的一致性（即 Drift Check）。
- **从 Spec 自动生成测试用例**：让 `implementer` Agent 根据 Spec 中的验收场景，自动生成对应的单元测试或集成测试骨架。

**参考资源**：
- GitHub Spec Kit：`github/spec-kit`
- LightSpec：轻量级规格工作流，适合快速试点
- 推荐阅读 GitHub 开源的 Spec Kit 官方博客，了解 spec-first 工作流的设计理念

#### 方向二：引入 MCP——用标准化工具替代自然语言编排（优先级：中，中期可实现）

当前 Agent 之间通过自然语言进行任务传递，存在效率损失和语义歧义。**MCP (Model Context Protocol)** 提供了更高效的替代方案——将任务封装成标准化的“工具”，Agent 通过结构化数据而非自然语言调用。

**典型应用场景**：
- **GitLab API 交互 MCP**：将 `/git-merge` 中的 API 调用封装为 MCP 工具，Agent 直接调用 `create_merge_request` 而非通过 curl 命令。
- **ONES 单查询 MCP**：封装 ONES API，让 Agent 自动获取需求/Bug 详情，无需用户手动复制。
- **Spec 同步 MCP**：封装 OpenSpec 的 proposal/apply/archive 操作，让 Agent 以工具形式操作规范。

**参考资源**：
- `github-mcp-server`：官方 GitHub MCP 服务器实现
- `cyanheads/git-mcp-server`：通用 Git MCP 服务器
- `spec-driven-dev-mcp`：将 SDD 工作流引入 AI IDE 的 MCP 服务器

**实施路径**：建议先从最频繁调用的 GitLab API 入手，封装一个最小可用 MCP，验证效果后再逐步扩展。

#### 方向三：上下文工程持续优化——将 AI 产出可靠性纳入系统保障（优先级：持续，长期坚持）

上下文工程（Context Engineering）是 2025 年兴起的核心理念，核心是为 AI 代理设计和管理信息，使其可靠地产出预期结果。

**具体实践方向**：

- **规则分层与按需加载**：利用 `.claude/rules/` 的 `paths` 字段，实现“修改 API 代码时自动加载 API 规范，修改前端代码时不加载”，精准控制上下文大小。
- **决策日志与可追溯性**：让每个 Agent 在执行关键操作（如分支创建、MR 描述生成）时，输出决策依据，形成可审计的操作记录。
- **Explore → Plan → Execute 方法论固化**：将这一三阶段方法论融入 `orchestrator` Agent 的调度逻辑，确保复杂任务先探索、再规划、后执行。
- **人机协作界面优化**：在关键节点设置 HITL（Human-in-the-Loop）确认点，平衡自动化效率与人工控制权。

**参考资源**：
- Context Engineering (Forrester, 2025)：系统阐述上下文工程的核心理念与最佳实践
- Explore, Plan, Execute 方法论：三阶段 AI 辅助开发方法
- Anthropic 官方 Subagents 最佳实践文档（Mintlify）：包含 Agent 与 Skill 结合使用的实战示例

### 9.3 演进路线图总览

```
当前（已验证）                 →   短期（1-2 月）          →   中期（3-6 月）          →   长期（持续）
┌─────────────────────────┐     ┌─────────────────────┐     ┌─────────────────────┐     ┌─────────────────────┐
│ 三层 Agent 架构          │     │ 深化 SDD 集成        │     │ 引入 MCP             │     │ 上下文工程持续优化    │
│ - Orchestrator          │ ──→ │ - Spec Kit 自动化    │ ──→ │ - GitLab API MCP     │ ──→ │ - 规则分层加载       │
│ - Analyzer/SpecWriter/  │     │ - Spec 自动验证      │     │ - ONES 查询 MCP      │     │ - 决策日志审计       │
│   Implementer           │     │ - 测试用例自动生成   │     │ - Spec 同步 MCP      │     │ - HITL 界面优化      │
│ - 原子能力层            │     └─────────────────────┘     └─────────────────────┘     └─────────────────────┘
└─────────────────────────┘
```

每个阶段都以前一阶段的稳定运行为基础，逐步叠加新的能力，而非一次性推倒重来。建议在完成当前架构的全面验证后，再启动下一阶段的探索。
